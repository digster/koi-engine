# 2026-07-01 — Step 11: Multiple lights (directional + point + spot)

## What we did

Completed **Step 11**. The single hardcoded directional "sun" became a **list of lights** of
three kinds that the fragment shader loops over and accumulates: a **directional** sun, coloured
**point** lights (fade with distance), and a **spot** light (a cone). User chose (clarifying
question) **multiple lights** over PBR / normal mapping as the next slice of the roadmap's
"Step 11+" lump.

## Key decisions

- **Additive lighting in a shader loop.** Kept the Step 7 Blinn-Phong math per light; wrapped it
  in a loop over a fixed array, summing contributions. **Ambient added once**, up front. The
  **vertex shader is unchanged** (already outputs world pos + normal).
- **Two new concepts, with pure CPU twins.** `attenuation` = windowed inverse-square
  `clamp(1-(d/range)^4,0,1)^2 / (d^2+1)` (UE4-style; exact 0 at `range` → finite radius).
  `spotFactor` = `smoothstep(outerCos, innerCos, dot(L,aim))` — compares **cosines** (no `acos`);
  remember cos decreases with angle so inner cutoff cos > outer. Both live in `scene/Light.hpp`
  and are unit-tested; the shader stays runtime truth (same pattern as `PostProcess.hpp`).
- **Fixed `MAX_LIGHTS=8` std140 array** in the light UBO (`set=3,b=0`). Uniform layout is fixed at
  pipeline-build, so no dynamic size. Each `GpuLight` packed as **exactly 4×vec4 (64 B)** so the
  C++ struct in `recordScene` and the GLSL struct agree byte-for-byte (a stray `vec3` would shift
  the whole array — same alignment-risk class as Step 9's `--msl-decoration-binding` bug).
- **Engine owns the lights** (`std::vector<Light>`), like the camera + `PostSettings`; threaded to
  the renderer as `std::span<const Light>`. Not scene-graph nodes yet (transform inheritance is a
  noted future refinement).
- **Shadows stay on the sun only** (light 0). Each caster needs its own map (a point light → a
  cube map), so we keep the one Step-9 shadow map, now driven by `lights[0].direction`, and the
  shader applies the shadow factor only to light 0 when it's directional. If the sun is toggled
  off it's skipped in packing, so index 0 becomes a non-directional light → no shadow, correctly.

## What was built / changed

- **`scene/Light.hpp`** (new): `LightType` enum, `Light` struct (type/pos/dir/color/intensity/
  range/inner+outerCutoffCos/enabled), `MAX_LIGHTS`, pure `attenuation`/`spotFactor`/`smoothstep01`/
  `activeLightCount`.
- **`shaders/triangle.frag`**: single light → `GpuLight lights[MAX_LIGHTS]` + `lightCount` in the
  UBO; loop with per-type `L`/attenuation + spot cone; shadow gated to `i==0 && type==0`. Sampler/
  UBO counts unchanged → no `loadShader` edit. Recompiled GLSL→SPIR-V→MSL cleanly.
- **`GpuRenderer.{hpp,cpp}`**: `lights` param added to `renderFrame`/`captureFrame`/
  `renderSceneAndPost`/`recordScene`; `recordScene` packs enabled lights into the std140
  `LightUniform` (ambient, cameraPos, lightViewProj, `Sint32 lightCount[4]`, `GpuLight[8]`).
  `sunDirection()` free fn removed; `computeLightViewProj(sunDir)` now takes the sun dir;
  `renderSceneAndPost` pulls it from `lights[0]` (with a `kFallbackSunDir`).
- **`Engine.{hpp,cpp}`**: `std::vector<Light> lights_` + `lightOrbit_`; new `setupLights()` (sun +
  warm point + cool orbiting point + overhead green spot). Orbits `lights_[2]` in `run()`. Keys
  `5`/`6`/`7` toggle point lights / spot / sun; controls log updated. Passes `lights_` to both
  render calls. `<cmath>` for `std::cos/sin`.
- **`tests/test_light.cpp`** (new, registered in CMake): attenuation (1 at d=0, monotone, 0 at
  range), spotFactor (1 inside / 0 outside / smooth + monotone across the edge), activeLightCount
  (counts enabled, caps at MAX_LIGHTS).
- **`main.cpp`**: title → "Step 11: Multiple lights".
- **Docs**: new `docs/12-multiple-lights.html` (concepts: 3 light types, attenuation, spot cone,
  additive loop, std140 fixed array, shadows-on-sun-only, deferred/clustered as future); nav link
  added to all 13 pages; index card + roadmap (Step 11 ✅, 12+ = PBR/more shadows). README (status,
  controls, doc list, roadmap) + ARCHITECTURE (Light row, data-flow, new §, planned layering).
