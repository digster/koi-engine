# 2026-07-03 — Step 15: Image-based lighting (IBL)

## What we did

Completed **Step 15**. The Step 14 **skybox now lights the scene** instead of just sitting behind it —
the fix for Step 12's **dark metals** (a metal has no diffuse, so with only a flat ambient + direct
lights it reflected almost nothing). User picked (clarifying question) **full IBL** (diffuse + specular
split-sum) in one step, baked via **GPU precompute at startup** (render passes, not compute). First
code milestone since Step 14.

## Key decisions

- **Split-sum approximation** (Karis/Epic 2013). Three maps baked from the skybox cubemap:
  - **diffuse irradiance** cube (32², cosine-convolved hemisphere) — sampled by the normal.
  - **specular prefilter** cube (128², 5 roughness **mips**; mip = roughness via `textureLod`) — GGX
    importance-sampled; samples the env's own mips by pdf to kill fireflies.
  - **BRDF LUT** (512², RG16F) — the split-sum's **environment-independent** factor, so baked ONCE in
    `createIblResources` (never re-baked); the two cubes bake in `bakeIbl` from whatever sky is loaded.
- **Bake by rendering into cube faces** — `bakeIbl` draws the cube six times per target (once per face,
  and per mip for the prefilter) with a 90° camera down each axis (`ibl_cube.vert`, `cubeCaptureViews`).
  `SDL_GPUColorTargetInfo.layer_or_depth_plane` + `.mip_level` select the subresource. **No `Mat4`
  inverse needed** (the gap Step 14 hit) — the cube corner's object-space position IS the world sample
  direction, so `lookAt`+`perspective` suffice.
- **Bake-at-load, not per-frame.** `bakeIbl` is called at the end of `loadCubemap` (a sky reload
  re-bakes automatically). `iblReady_` gates use until a bake ran; `iblEnabled_` (key **9**) is the
  runtime A/B toggle, passed to the shader in the light UBO's spare **`ambient.w`** lane (no UBO layout
  change). Shader uses a **uniform if-branch** on `ambient.w` (IBL vs flat fill) — safe for the
  cubemap's implicit-derivative sampling, and avoids NaN-from-unbaked-maps.
- **IBL geometry term uses `k = roughness²/2`**, NOT the direct-lighting `(r+1)²/8` — a classic silent
  IBL bug, so it's a separate function (`geometrySmithIBL`) in both the shader and the CPU mirror.
- **Demo showcase:** the smooth metal sphere (roughness lowered 0.20→0.08). Already metallic, so IBL's
  effect is immediate: dark ball → mirrors the sky.

# 2026-07-03 — Step 16: glTF PBR material import (Damaged Helmet)

## What we did

Completed **Step 16**. Materials are now **imported from glTF files** instead of hand-authored in
`Engine::buildScene`. This pays off the Step 13 IOU (that step built the per-pixel map machinery but fed
it generated BMPs and deferred real glTF import + PNG loading + the Damaged Helmet). User picked (via
clarifying questions) to do it **with libraries like the OBJ path** (cgltf already present + add
stb_image), to **include emissive**, and to **fix sRGB** for colour textures. First code milestone since
Step 15 (same day).

## Key decisions

- **cgltf (already parsing geometry) + stb_image (new).** cgltf parses the glTF material but does NOT
  decode images — it hands us the compressed PNG/JPG bytes. Added **stb_image** as a third single-header
  dep, fetched via `koi_fetch_header` and **pinned to a commit** (stb has no tags), implementation
  `#define`d once in the existing warnings-off `ModelLoader.cpp`. GpuRenderer.cpp includes the header
  decl-only for `loadTexture`'s PNG path.
- **`loadModel` now returns `LoadedModel { mesh, material }`** (was `shared_ptr<Mesh>`). glTF fills the
  material; OBJ leaves it null (`.mtl` out of scope). Only 2 call sites (Engine.cpp).
- **Material extracted before `cgltf_free`** — embedded images decode straight out of the still-live glTF
  buffers (`cgltf_buffer_view_data` gives an offset-applied pointer). Image resolution: `buffer_view`
  (embedded `.glb`) → `stbi_load_from_memory`; else `uri` (external) → `loadTexture(path, srgb)`; data-URI
  deferred. Albedo is guaranteed non-null (1×1 solid from `base_color_factor` if no texture) since
  `recordNode` dereferences it.
- **sRGB is a texture-FORMAT choice, not a pass.** `uploadToGpuTexture` gained `bool srgb` →
  `R8G8B8A8_UNORM_SRGB` (GPU un-gammas on sample). Colour maps (base-colour, emissive) pass true; DATA maps
  (MR/normal/AO) stay linear. Same bytes uploaded. New public `createTextureFromRGBA` (decoded-bytes entry,
  also `withMips` for the 1×1 solid).
- **Emissive = added, not reflected.** `Material` gained `emissive` + `emissiveFactor[3]` (default 0 ⇒ no
  glow, existing objects unchanged, no shader branch). `triangle.frag`: 9th sampler at **slot 8**, second
  `vec4 emissive` in MaterialUBO (still one uniform buffer), `color += texture(uEmissiveMap,uv).rgb *
  emissive.rgb` after shading → bright pixels feed Step 10 bloom. Pipeline `numSamplers` 8→9; `recordNode`
  binds slot 8 + pushes 8 floats.
- **Helmet asset downloaded at configure time** (`file(DOWNLOAD)`, **soft-fail** — WARNING not
  FATAL_ERROR, engine warns+continues if absent), gitignored (`assets/DamagedHelmet.glb`, ~3.6 MB). Placed
  as a hero node, rotated upright in Engine (we import raw primitive geometry, **ignoring glTF node
  transforms** — the asset is Z-up).

# 2026-07-03 — Step 17: Engine / app separation

## What we did

Completed **Step 17**, the roadmap's "defining production milestone" and the first *structural* (not
rendering) step. The demo that lived **inside** `Engine` moved out into a standalone **sample app**, so
`koi_core` is genuinely reusable. User picked (via clarifying questions): the **Application base-class**
API shape, and **`samples/demo/` → a `koi-demo` executable**. Third code milestone today (Steps 15, 16, 17).

## Key decisions

- **Inversion of control.** The frame loop stays in the engine (only it can acquire a swapchain image,
  pace, submit), but its *contents* are the app's. `Engine::run()` → **`bool run(Application&)`**, which
  calls back into the app's hooks. `Engine` now owns **no** `Node`/`Camera`/`Light`/`PostSettings` — only
  window, renderer, loop, delta-time, and the `KOI_CAPTURE`/`KOI_MAX_FRAMES` paths. Services exposed to the
  app: `renderer()` (out-of-line, so the header keeps forward-declaring GpuRenderer) + `requestQuit()`.
- **`koi::Application`** (`src/core/Application.hpp`, header-only): `onStart` / `onUpdate(dt)` /
  `onEvent` (default no-op) / `frameView()` / `onShutdown` (default no-op). The demo is `DemoApp`.
- **`koi::FrameView`** (`src/renderer/FrameView.hpp`, header-only): `{clearColor, view, root(const Node*),
  cameraPos, lights(span), post}` — the one value crossing the boundary each frame. `renderFrame`/
  `captureFrame` collapsed from the same six args to a **single `const FrameView&`** (bodies unchanged —
  aliased the fields at the top), unifying the live + headless paths behind one "what to draw".
- **Transform ownership → app.** `FrameView.root` is `const Node*` (read-only render contract), so the
  **app** owns `updateWorldTransforms()` — called in `DemoApp::onStart` (poses the t=0 capture) and at the
  end of `onUpdate`. The engine holds no `Node*`, so it can't (and shouldn't) run the transform pass.
- **Lifetime hazard.** App now owns GPU-backed scene, engine owns the device. `Engine::run` calls
  `app.onShutdown()` **while `renderer_` is still alive**; `DemoApp::onShutdown` resets `sceneRoot_` there,
  so meshes/textures free before `Engine::shutdown()` destroys the device. `Engine::shutdown` now only
  resets `renderer_`/`window_`.
- **Behaviour-preserving.** Pure refactor — the demo code moved almost verbatim (buildScene/setupLights →
  `DemoApp::onStart` using `engine.renderer()`; animation → `onUpdate`; the key-1–9 switch → `onEvent`,
  Esc → `engine.requestQuit()`).

# 2026-07-03 — Step 18: Quaternions

## What we did

Started from the user's open-ended "give me options to pick up next from the roadmap." Presented four
roadmap-grounded candidates (CI + golden images, transparency, quaternions, glTF scene/Sponza); user
picked **quaternions** — the math prerequisite the roadmap sequences before skeletal animation. Two scope
decisions (asked): **camera stays on yaw/pitch** (Transform-only), and an **internals-only demo** (no new
on-screen object). Replaced Euler-angle rotation with a hand-rolled unit quaternion.

## Key decisions

- **New `Quat` type**, header-only in `src/math/Quat.hpp`, glTF `(x,y,z,w)` layout (vector part first,
  scalar last) so future node-hierarchy import drops in without shuffling. `Quat.hpp` includes `Mat4.hpp`,
  not the reverse (Mat4 stays quaternion-unaware).
- **`Transform` stores `Quat rotation`** instead of `Vec3 rotationEuler`; `localMatrix()` = `translation *
  rotation.toMat4() * scaling`. T·R·S order unchanged — only the middle factor's source changed.
- **Behaviour-preserving via `Quat::fromEuler`**, which composes `qz*qy*qx` to match the old
  `rotationZ*rotationY*rotationX`. This is why the render stayed visually identical and existing
  Transform/Node tests kept passing.
- **Camera deliberately NOT converted.** Clamped yaw/pitch is the correct control scheme for an FPS/fly cam
  (two DOF, no roll, can't gimbal-lock in practice). Converting would add unwanted roll and buy nothing.
  Knowing when *not* to use quaternions is a documented teaching point.
- **Demo spin** (hub/spinner): a quaternion has no per-axis angle to `+=`, so `DemoApp` gained float
  accumulators `hubSpin_`/`spinnerSpin_` and rebuilds `rotation = Quat::fromAxisAngle({0,1,0}, angle)` each
  frame — exact same pure-Y spin, and rebuilding fresh avoids drift off unit length.
- **`rotate()`** uses the Euler–Rodrigues closed form (a few dot/cross products) rather than two literal
  quaternion multiplies; assumes a unit quaternion.
- **Gotcha fixed while writing:** `fromEuler` defined inside the struct couldn't see the free
  `operator*(Quat,Quat)` declared later (complete-class context only exposes *members* early, not
  later free functions); moved `fromEuler` out-of-line after the operator.
- **Deferred:** `Mat4` inverse; camera quaternions; wiring `slerp` into the demo (implemented + tested, but
  the visible payoff is saved for skeletal animation).

## What was built / changed
