# 2026-07-02 — ROADMAP.md production-readiness review (analysis only)

## What we did

The user stated a new long-term goal: eventually take Koi to a **professional level and use it for
production apps**. Asked for a review of `ROADMAP.md` from that perspective — **no changes yet**,
just the list of improvements. This session produced analysis only; ROADMAP.md, docs, and code are
untouched (only PROMPT.md + this memory file were written).

## Findings reported (candidate ROADMAP.md improvements)

1. **The defining milestone is missing: engine/app separation.** Today `Engine::buildScene`
   hardcodes the demo scene and `main.cpp` is the app. "Production apps" requires Koi to become a
   consumable library with a public API boundary, a `samples/` or sandbox app, and eventually
   versioning. Nothing on the roadmap names this.
2. **Flag "architecture pivots" separately from additive features** — things cheap to plan early
   and expensive to retrofit: multithreading/job system (absent entirely), render-queue extraction
   (decouple scene traversal from draw submission; prerequisite for sorting/culling/instancing
   already on the roadmap), shader variant/permutation management (the single uber
   `triangle.frag` grows per feature; normal-maps-optional + skinning + deferred will force it),
   and a memory strategy (everything is `shared_ptr`/`unique_ptr`; no allocator/streaming plan).
3. **Promote cross-platform validation + CI from "Platform & build" (far out) to near-term.**
   Only macOS/Metal is exercised; backend-specific bugs accumulate (the Step 9 spirv-cross
   `--msl-decoration-binding` swap is the existing proof). A Linux/Vulkan CI build (lavapipe) +
   `KOI_CAPTURE`-based **golden-image regression tests** is the highest-leverage production item
   and builds on infrastructure that already exists.
4. **Asset pipeline (offline cooking) missing** — runtime asset manager is listed, but not an
   offline import→process→pack step (engine-native formats, KTX2 fits here, `gen_models.py` is a
   seed).
5. **Robustness track missing:** defined failure policy (missing assets, device lost), fuzzing the
   OBJ/glTF loaders (untrusted input), sRGB/color-management correctness (albedo currently treated
   as linear — documented shortcut).
6. **Roadmap-document structure:** add a "definition of 1.0" / quality-gates section; tag items by
   phase (learning → hardening → production); fixed-timestep loop should precede physics; small
   duplication — "asset hot-reload" appears in both Scene & data architecture and Platform & build.

## Next

phase tags), add the engine/app-separation milestone, and move CI + golden-image testing up.

# 2026-07-02 — Step 13: Texture maps + normal mapping + mipmaps/anisotropy

## What we did

Completed **Step 13**. Material parameters now vary **per pixel** from texture maps, plus fine
surface relief via **normal maps**, plus **mipmaps + anisotropic filtering**. User picked the fuller
slice (clarifying question): **"+ metallic/roughness/AO maps"** (not normal-map-only, not the full
glTF-import/Damaged-Helmet scope). This was the first code milestone since Step 12 (PBR).

## Key decisions

- **Fallback textures instead of shader permutations.** Renderer owns two 1×1 maps — `whiteTex_`
  (255,255,255) and `flatNormalTex_` (128,128,255 → tangent (0,0,1)). A material omitting a map binds
  the fallback, so the shader math reduces *exactly* to Step 12 (flat normal ⇒ geometric normal;
  white MR ⇒ scalar factor; white AO ⇒ 1). **No `if`/uniform flag** → dodges the shader-variant pivot.
- **glTF metallic-roughness packing** (G=roughness, B=metallic), one texture; AO separate (R). The
  Step-12 `metallic`/`roughness` scalars survive as **factors** multiplying the sampled channels — so
  the material uniform layout / `loadShader` UBO count are **unchanged**; only sampler count changed.
- **vec3 tangent, bitangent = N×T in-shader** (Gram-Schmidt orthonormalize). Simpler than a signed
  vec4; mirrored-UV handedness is the documented simplification.
- **5 fragment samplers, per-material maps first:** set=2 → b0 albedo, b1 metalRough, b2 normal,
  b3 AO (bound per draw in `recordNode`), b4 shadow (moved from b1, bound once in `recordScene`).
  MSL still uses `--msl-decoration-binding`, so slot↔binding holds (Step 9 class of bug).
- **Verify on our own meshes, not Damaged Helmet.** Helmet needs glTF material import + PNG loading
  (a roadmap-separate step). Deferred explicitly.

## What was built / changed

- **`src/renderer/Vertex.hpp`**: +`float tangent[3]` (sizeof 44→56, offset 44); static_asserts +
  `tests/test_vertex.cpp` updated.
- **`src/renderer/Tangents.hpp`** (new, SDL-free): `triangleTangent` (Lengyel from pos+UV),
  `orthonormalizeTangent` (Gram-Schmidt + safe fallback), `anyPerpendicular`. Unit-tested in new
  **`tests/test_tangent.cpp`** (6 cases; registered in CMake).
- **`src/renderer/Primitives.cpp`**: hardcoded per-face tangents (cube +X/+Z/-X/-Z per face; plane +X).
- **`src/renderer/ModelLoader.cpp`**: `makeVertex` +tangent params; `computeTangents` helper; glTF
  reads `cgltf_attribute_type_tangent` (vec4→xyz) if present else computes; OBJ always computes.
- **`src/scene/Material.hpp`**: rename `texture`→`albedo`; +optional `metalRough`/`normalMap`/`ao`
  (maps go LAST so positional inits still compile); scalars documented as factors.
- **`src/renderer/GpuRenderer.{hpp,cpp}`**: `uploadToGpuTexture(..., bool withMips=true)` (num_levels,
  `SAMPLER|COLOR_TARGET`, `SDL_GenerateMipmapsForGPUTexture` after the copy pass);

# 2026-07-02 — Step 14: Skybox & environment cubemap

## What we did

Completed **Step 14**. The scene is now wrapped in a **cubemap sky** drawn behind the geometry —
the first cubemap in the engine, and the deliberate groundwork for Step 15 (IBL, which reuses the
cubemap plumbing to light surfaces from the environment). User picked (clarifying question) the
**cube-mesh** technique over fullscreen-triangle, and a **day-sky** look. First code milestone since
Step 13.

## Key decisions

- **Cube-around-the-camera, not fullscreen-triangle.** Reuses `makeCubeMesh` (position attribute
  only, at the full 56-byte `Vertex` stride — the shadow-pipeline precedent). The cube's object-space
  corner *is* the sample direction, so no ray reconstruction and — crucially — **no `Mat4` inverse**
  (the math library has none; the fullscreen variant would need it). Canonical beginner intro.
- **Three rendering tricks for "behind everything, infinitely far":** (1) strip the view's
  **translation** (zero the last column) → `skyViewProj = projection * viewNoTranslation`, so the sky
  turns but never slides; (2) `gl_Position = clip.xyww` pins post-divide depth to 1.0 (far plane);
  (3) pipeline tests **`LESS_OR_EQUAL`** with depth-write **off**, and the sky is drawn **last** in
  `recordScene` → fills only background pixels, no overdraw of shaded geometry.
- **Dedicated CLAMP_TO_EDGE cubemap sampler** (the shared `sampler_` REPEATs → would seam at face
  edges). New `skyboxSampler_`.
- **Renderer owns the sky** (like `shadowMap_`): `skyboxCubemap_`/`skyboxSampler_`/`skyboxPipeline_`
  + `skyboxMesh_`. The Engine calls `loadCubemap` with asset paths (paths stay in the app layer). A
  runtime `skyboxEnabled_` flag (key `8`) gates the draw; the draw is also skipped if no cubemap.
- **Lifetime subtlety:** `skyboxMesh_` is a `shared_ptr<Mesh>` *member*; member dtors run after
  `~GpuRenderer`'s body (device already destroyed), so the dtor **explicitly `.reset()`s it** (and
  releases the cube texture/sampler/pipeline) while the device is still alive.
- **The tuning story (cross-system gotcha):** the first sky washed out to white because its
  brightness crossed Step 10's **bloom threshold (0.9)** → the whole horizon bloomed. Fix: deepen the
  palette (sub-bloom sky body) and set `kSkyIntensity = 1.25`, leaving only the **sun disk** above the
  threshold so just it glows. Caught via `KOI_CAPTURE` (only visible in a rendered frame).

## What was built / changed

- **`shaders/skybox.vert`** (new): forwards `inPosition` as `vDir`; `gl_Position = (skyViewProj *
  vec4(inPosition,1)).xyww`. One vertex UBO (`set=1,b=0`).
- **`shaders/skybox.frag`** (new): `samplerCube uSky` (`set=2,b=0`); `texture(uSky, normalize(vDir)) *
  kSkyIntensity` (1.25). No fragment UBO.
- **`CMakeLists.txt`**: `koi_add_shader(skybox.vert/.frag)` (compile GLSL→SPIR-V→MSL).
- **`tools/gen_skybox.py`** (new; PEP 723, `uv run`): six 256² day-sky BMP faces
  (`sky_{px,nx,py,ny,pz,nz}`), generated by **inverting the standard cube-face direction convention**
  per texel then evaluating an analytic sky (zenith→horizon gradient + ground + sun disk at
  `-lights[0].direction`). Seamless because generation and GPU sampling share the direction convention.
- **`src/renderer/GpuRenderer.{hpp,cpp}`**: new members + `createSkyboxResources()` (pipeline: 1
  position attr, `LEQUAL`, no depth write, HDR color + depth targets; CLAMP sampler; cube mesh),
  `uploadCubemap()` (CUBE, 6 layers, per-face copy region, mip-gen), `loadCubemap()` (6 BMPs →
  `uploadCubemap`), `recordSkybox()` (translation-strip + bind + draw, called at end of `recordScene`),
  destructor releases, `isValid()` extended.
- **`src/core/Engine.cpp`**: `buildScene` calls `loadCubemap({assets/sky_*.bmp})` (warns + continues on
  failure); `SDLK_8` toggles the sky; controls log updated.
- **Docs**: new `documentation/docs/15-skybox-and-cubemaps.html` (concepts: cubemaps sampled by
  direction, cube-around-camera, translation-strip, far-plane depth trick, seams/CLAMP, the bloom-
