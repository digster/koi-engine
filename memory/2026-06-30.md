# 2026-06-30 — Step 9: Model loading (OBJ + glTF) & shadow mapping

## What we did

Completed **Step 9**, a large two-part iteration (the user chose to bundle them):
1. **Model loading** from files via single-header libraries (per the user: use a library
   for parsing, spend the from-scratch effort on graphics). **tinyobjloader** for `.obj`,
   **cgltf** for `.glb/.gltf`. A sphere (glb) and torus (obj) now render beside the cubes.
2. **Shadow mapping**: a depth pre-pass from the light's view into a sampleable shadow map,
   sampled in the main pass so every object casts a real shadow on the floor.

## Key decisions

- **Library, not hand-rolled** (user's call): tinyobjloader + cgltf, downloaded at configure
  time like `doctest` (single headers into `build/_deps/models`). The one TU that `#define`s
  their `*_IMPLEMENTATION` (`ModelLoader.cpp`) is compiled `-w` (warnings off — third-party).
- **OBJ dedup + 32-bit indices**: OBJ indexes v/vt/vn separately → dedup each unique triple
  into one `Vertex`. Loaded models can exceed 65 535 verts → `Mesh` stores a
  `SDL_GPUIndexElementSize`; `createMesh` gained a `Uint32` overload; `recordNode` binds with
  the stored size.
- **Models generated procedurally** (no copyright): `gen_models.py` (uv+numpy+pygltflib)
  writes `assets/sphere.glb` (UV sphere) and `assets/torus.obj`, both with pos/normal/uv.
- **Shadow mapping = two passes**, single directional light. `renderShadowPass` (depth-only
  pipeline, `shadow.vert` + empty `shadow.frag`, 0 color targets) renders depth into a 2048²
  depth texture (`DEPTH_STENCIL_TARGET | SAMPLER`). Main fragment shader projects worldPos by
  `lightViewProj`, samples the map, compares with slope-scaled **bias** + **2×2 PCF**;
  occluded fragments keep ambient, lose diffuse+specular. `Mat4::orthographic` added (the
  light's frustum); `lightViewProj` computed once and used by BOTH passes.

## Bug fixed (worth remembering)

With **two** fragment samplers (material texture + shadow map), `spirv-cross` ordered the MSL
texture/sampler indices by **usage order** (shadowFactor is defined first → uShadowMap got
index 0), not by `layout(binding=)`. SDL binds by slot, so they were **swapped** on Metal — the
surface texture sampled the depth map → whole scene washed to grayscale. Fix: compile MSL with
**`--msl-decoration-binding`** so indices follow the GLSL bindings. The **capture** caught it
(invisible to the compiler). This is the same class of risk flagged since Step 6.

# 2026-06-30 — Step 10: Post-processing (HDR offscreen target + fullscreen chain)

## What we did

Completed **Step 10**. The scene no longer renders straight to the swapchain — it renders into
an engine-owned **off-screen HDR target**, then a chain of **fullscreen passes** processes it
before present. User chose (clarifying question) to include **all** candidate effects, so we
built the full stack: **bloom**, **ACES tone-mapping** + exposure + **gamma**, **vignette**,
and **FXAA**. Every effect toggles at runtime.

## Key decisions

- **HDR is the enabling choice**: scene target is `R16G16B16A16_FLOAT` so highlights exceed 1.0
  — gives tone-mapping range to compress and bloom real bright areas. An 8-bit target would clip
  both. `isValid()`/`createTrianglePipeline` verify the format is a sampleable color target.
- **Fullscreen triangle, not a quad**: one oversized triangle from `gl_VertexIndex`, **no vertex
  buffer** (`fullscreen.vert`; draw 3 verts). Shared by all four post pipelines.
- **Pass order is dictated by colour space**: bloom reads HDR-linear (before tone-map); composite
  applies exposure→ACES→vignette→**linear→sRGB gamma**; **FXAA runs last** on the gamma-encoded
  LDR image (it judges edges by perceived luma). Getting this order wrong is a classic bug.
- **Bloom at half-res**, separable Gaussian, ping-pong `bloomA_/bloomB_` (`kBloomBlurPasses = 4`).
- **No-drift refactor**: one `renderSceneAndPost` (shadow → scene-into-HDR → `runPostChain`) used
  by BOTH `renderFrame` (→swapchain) and `captureFrame` (→capture target), so the BMP matches the
  window. Intermediate targets use `cycle=true` (same trick as the shadow map); the swapchain does
  not. Effect-off = cheap pass-through (bloom intensity 0 / FXAA copy / tonemap clamp).
- **gamma now applied** (was missing pre-Step-10): the engine previously output linear values to
  an sRGB display, so captures were darker; composite now encodes. Input textures still treated as
  linear (documented shortcut).

## What was built / changed

- **Shaders (5 new)**: `fullscreen.vert`, `bright.frag` (luma threshold, soft knee), `blur.frag`
  (9-tap separable Gaussian, direction uniform), `composite.frag` (exposure, ACES, vignette,
  gamma), `fxaa.frag` (compact FXAA). Registered in CMake.
- **`renderer/PostProcess.hpp`** (new, SDL-free): `PostSettings` (exposure, bloom threshold/
  intensity, per-effect bools) + pure helpers `halfExtent`/`luminance`/`acesToneMap`.
- **`GpuRenderer`**: `sceneHdr_` + `ldr_` + `bloomA_/bloomB_` targets, `postSampler_` (linear
  clamp), 4 fullscreen pipelines; `createPostResources`, `buildFullscreenPipeline`,
  `createColorTarget`, `ensureSceneTargets`, `runPostChain`, `renderSceneAndPost`, `beginColorPass`,
  `bindPostSampler`, `drawFullscreen`. Main pipeline colour format swapchain→HDR. dtor/isValid
  updated. `renderFrame`/`captureFrame` take a `PostSettings` and delegate to `renderSceneAndPost`.
- **`Engine`**: `PostSettings post_` member; keys `1`–`4` toggle tonemap/bloom/FXAA/vignette,
  `[`/`]` adjust exposure; controls log updated. Passes `post_` to both render calls.
- **Tests**: `tests/test_post.cpp` (halfExtent clamps, luma weighting, ACES 0→0/bounded/monotone).
- **Docs**: `docs/tuts/11-post-processing.html`; nav link added to all pages; index card + roadmap
  (Step 10 ✅, 11+ = lights/PBR); README + ARCHITECTURE (GpuRenderer + PostProcess rows, frame
  data-flow, new Step 10 section, planned layering).
