# 2026-07-13 — VS Code + clangd setup (tooling, not a Step)

## What we did

Wired the repo up for **VS Code using the clangd extension** instead of the default
Microsoft C/C++ (`ms-vscode.cpptools`). clangd reads the exact per-file compile
flags from `build/compile_commands.json` (CMake already exports it via
`CMAKE_EXPORT_COMPILE_COMMANDS ON`, and the root `.clangd` already points clangd at
`build/`), so editor diagnostics/navigation match the real build. The two
IntelliSense engines conflict, so the setup actively suppresses cpptools.

User decisions (asked up front): **commit** the config to the repo, and do the
**full dev setup** (clangd + build tasks + LLDB debug), not just the clangd minimum.

### Files added — `.vscode/` (all committed)
- **`extensions.json`** — recommends `llvm-vs-code-extensions.vscode-clangd` and
  `vadimcn.vscode-lldb` (CodeLLDB — the clangd-friendly debugger on macOS, since we
  don't use cpptools' bundled debugger); `unwantedRecommendations` lists
  `ms-vscode.cpptools` so VS Code prompts to disable the rival engine.
- **`settings.json`** — `clangd.path = /usr/bin/clangd` (Apple clangd 21, exact match
  to `/usr/bin/c++`); `clangd.arguments`: `--background-index`, `--clang-tidy`,
  `--completion-style=detailed`, `--header-insertion=never`,
  **`--query-driver=/usr/bin/c++`**, `--pch-storage=memory`;
  `C_Cpp.intelliSenseEngine = disabled`; `files.associations` (`*.hpp`→cpp,
  `*.vert`/`*.frag`→glsl); `editor.formatOnSave = false`.
- **`tasks.json`** — configure/build/test via the existing `debug` CMake preset
  (`cmake --preset debug`, `cmake --build --preset debug` as default build task with
  `$gcc` problem matcher, `ctest --preset debug`).
- **`launch.json`** — CodeLLDB (`type: lldb`) launch configs for `build/koi-demo` and
  `build/koi-tests`, `cwd = ${workspaceFolder}` (matches documented `./build/koi-demo`
  run), `preLaunchTask` = the build task, commented `KOI_CAPTURE` env example.

### `.gitignore`
Changed `.vscode/` (ignore everything) → `.vscode/*` + negation exceptions for the

## Why `--query-driver` is the macOS linchpin

clangd, for security, won't run a compiler to discover its built-in/SDK header paths
unless that driver is explicitly allow-listed. Apple Clang keeps `<stddef.h>`, SIMD
intrinsics, and the macOS SDK path *inside the driver* (not in
`compile_commands.json`), so without `--query-driver=/usr/bin/c++` clangd reports
spurious "file not found" on system headers. `/usr/bin/c++` is exactly the compiler
recorded in the compilation database.
