HUD & Text

Goal of Step 23: put text on the screen. Everything the engine has drawn so far lives in the 3D world — meshes, lights, even debug lines float in world space and move with the camera. A HUD ("heads-up display") is different: it is pinned to pixels on the window — an FPS counter in the corner, a controls legend, a value readout — and it never moves. Building one means learning three new ideas: how to store letter shapes (a bitmap font in a texture atlas), how to place things in screen space instead of world space, and why user interface is drawn in its own pass, last, straight onto the finished image.

It builds directly on debug draw: the same immediate-mode "collect this frame's geometry, upload it, draw it once" shape — only now the geometry is textured triangles instead of coloured lines.

1. A GPU has no idea what a letter is

A graphics card can do exactly one thing: fill triangles with pixels, optionally reading colours from a texture. It has no concept of an "A" or a font. So to draw text we must give it letter shapes as something it understands — and the simplest possible shape is a tiny picture. That is a bitmap font: each character is a small grid of on/off pixels. Koi uses the classic 8×8 size, so one glyph is an 8-pixel-wide, 8-pixel-tall image — stored as just 8 bytes, one byte per row, one bit per column. A 1 bit is ink, a 0 is background.

the letter 'A', 8 bytes (one per row):     bits, MSB→LSB, drawn as pixels:
  0x0C  0x1E  0x33  0x33                       ..XX....   .XXXX...   XX..XX..   XX..XX..
  0x3F  0x33  0x33  0x00                       XXXXXX..   XX..XX..   XX..XX..   ........

We embed this table (the public-domain font8x8_basic) directly in the engine, in src/renderer/Font.hpp. No .ttf file, no font library — Koi stays fully offline and you can read a glyph at the bit level. (Bitmap fonts have a limit: they only look crisp at whole-number sizes. The modern alternative, a signed-distance-field font, stays smooth at any scale — a natural follow-up step.)

2. The texture atlas: one image, many glyphs

A shader can't sample a "byte array"; it samples a texture. So at start-up the renderer walks the font table and paints every glyph into one image — the font atlas — on a fixed grid of 8×8 cells. Drawing a character then becomes: draw a little quad, and point its texture coordinates (UVs) at that character's cell. A texture atlas is simply "many small images packed into one big one", and it exists because switching textures is expensive: with every glyph in a single atlas, a whole paragraph draws from one texture in one call.

the atlas: a 16 × 6 grid of 8×8 cells (128 × 48 texels)
  cell 0 = ' '   cell 1 = '!'   …   cell 33 = 'A'   …   cell 95 = ■ (solid white)
  a glyph paints WHITE where a bit is set, TRANSPARENT elsewhere → the alpha IS the coverage

Notice cell 95: a solid white square we reserve on purpose. A filled HUD panel (the translucent backing behind the text) is just a quad — and if it samples that all-white cell, it goes through the same textured pipeline as the glyphs. One pipeline, one draw, both text and boxes. Because a set bit is stored as opaque white and everything else as transparent, the fragment shader's whole job is atlasColour × tint: the tint supplies the colour, and the atlas's alpha supplies the letter's coverage.

3. Screen space: turning pixels into clip space

World geometry reaches the screen through the MVP matrix and perspective. The HUD skips all of that: its vertices are already where we want them, in pixels, with the origin at the top-left (x grows right, y grows down — how everyone thinks about a 2D screen). All that's left is to convert a pixel coordinate into clip space, the [−1, +1] square the GPU rasterises into. That conversion is an orthographic (no perspective) projection, and it's short enough to write by hand instead of building a matrix:

ndc.x =      px / width  * 2 − 1        // 0..width   →  −1..+1
ndc.y = 1 − (py / height * 2)           // 0..height  →  +1..−1   (flip: screen-y is DOWN, clip-y is UP)

The Y flip is the one subtlety: pixel (0,0) is the top-left, but clip-space +1 is the top, so we invert. The window's size is handed to the vertex shader as a uniform each frame, so the very same HUD geometry lands correctly at any window size, and resizing "just works". This lives in shaders/hud.vert.

4. Where UI is drawn: a final pass, in LDR

Here is the most important idea in this step, and where the HUD deliberately differs from debug draw. The debug lines are drawn inside the scene pass, into the HDR target — so they pass through tone-mapping, bloom and FXAA on the way to the screen. That's fine for a wireframe. It would be wrong for text: tone-mapping would shift its colour and FXAA would blur its crisp edges. So the HUD is drawn in its own pass, after the whole post-processing chain, straight onto the final LDR image.

scene → HDR target → [ bloom → tone-map → FXAA ] → final image → ● HUD pass ● → present
        (debug lines live in HERE)                                 (text lives HERE, last)

Three settings make that final pass behave like an overlay:

  • Load, don't clear. The pass uses LOAD_OP_LOAD — it keeps the finished frame that's already in the target and draws on top of it, rather than clearing it to a background colour.
  • Alpha blending on. The same "over" operator as the transparent pass, so a glyph's covered texels composite over the scene and its transparent texels vanish — and a panel with alpha < 1 lets the scene show through.
  • No depth at all. A flat 2D overlay has no depth to test; what sits on top is decided purely by draw order (we draw the panel first, then the text over it).

Because this pass targets the same image whether we're rendering to the window (renderFrame) or to a BMP (captureFrame), the HUD shows up in KOI_CAPTURE frames too — which is exactly how the screenshot for this page was made.

5. The collector: immediate-mode textured quads

The CPU side, src/renderer/Hud.hpp, is the twin of DebugDraw — pure (no GPU, no SDL) and immediate mode (rebuilt every frame). It turns two requests, text and rectangle, into a flat list of textured vertices:

struct HudVertex { Vec2 pos; Vec2 uv; Vec4 color; };   // pixel position, atlas UV, RGBA tint

hud.text(x, y, scale, colour, "FPS 60")   → one 6-vertex quad per character (two triangles)
hud.rect(x, y, w, h, colour)              → one 6-vertex quad sampling the white cell

Each character advances the "pen" by one cell (8×scale pixels — a fixed-width font), and a \n returns the pen to the start x and drops a line. Being pure, the whole thing is unit-tested headlessly in tests/test_hud.cpp: a wrong pen advance, a flipped quad, or a glyph pointing at the wrong cell would otherwise be a subtle "the text looks off" bug that no compiler catches.

6. One detail that bites: atlas bleeding

The atlas packs its 8×8 cells edge-to-edge, with no gap between them. With nearest-neighbour filtering (which we use, to keep the bitmap pixels sharp), a quad sampled exactly at a cell boundary can round into the neighbouring cell and drag in a stray pixel — a descender like g or p bleeding into the glyph below it. The fix is a half-texel inset: nudge each glyph's UV rectangle inward by half a texel on every side, so every sample stays strictly inside the intended cell while still covering all eight pixels (see cellUV() in Font.hpp). It's a classic atlas gotcha, worth knowing the moment you pack anything into a shared texture.

7. In the engine

The pieces mirror debug draw one-for-one:

  • Font.hpp — the embedded 8×8 glyph table + atlas layout, shared by the collector (for UVs) and the renderer (for baking the atlas), so the two can never disagree.
  • Hud.{hpp,cpp} — the pure collector: text(), rect(), clear(), vertices().
  • shaders/hud.vert / hud.frag — pixel→clip-space in the vertex shader; atlas × tint in the fragment shader.
  • GpuRenderer — createHudResources() bakes the atlas into a texture, makes a nearest-filter sampler, and builds the textured/blended/depth-less pipeline; uploadHud() and recordHud() upload this frame's vertices and draw them in the final overlay pass.
  • FrameView.hudVertices — the app hands its HUD geometry to the renderer through the same one-frame bundle that already carries the scene, lights and debug lines.

The demo (samples/demo/DemoApp.cpp) fills a Hud each frame in buildHud() — a translucent panel plus the engine title, a smoothed FPS/frame-time readout, the camera position, the current debug-toggle states, and a controls legend — and toggles it with the H key.

8. Honest limitations

This is a deliberately minimal first cut. The font is fixed-width (every glyph occupies one 8-pixel cell, so an i is as wide as an M) and looks best at whole-number scales. There is no clipping: text drawn past the window edge simply runs off (a real UI would set a scissor rectangle). And it covers only printable ASCII — no Unicode. Each of these is a clean future step (proportional/SDF fonts, a scissor rect, a text-layout pass); none of them change the core idea you now have.

9. Try it yourself

  1. Run it. cmake --build build && ./build/koi-demo, then press H to toggle the HUD. Watch the FPS and frame-time update live, fly around with WASD + mouse and see the camera position change, and toggle G / L / F to watch their states reflected in the readout.
  2. Capture a frame. KOI_HUD=1 KOI_CAPTURE=out.bmp ./build/koi-demo, then sips -s format png out.bmp --out out.png. The text stays pixel-crisp because it's drawn after tone-mapping and FXAA. (Without KOI_HUD the capture is byte-for-byte identical to Step 22 — the HUD adds nothing when nothing is queued, and its overlay pass is skipped entirely.)
  3. Read a glyph. Open src/renderer/Font.hpp and pick a character — trace its 8 bytes into an 8×8 grid of bits and you'll see the letter appear. That's the whole "font".

10. What's next

A frame profiler (a histogram of where each millisecond goes) needs it; so do ray-cast picking labels, an editor's panels, and eventually an in-engine console.
