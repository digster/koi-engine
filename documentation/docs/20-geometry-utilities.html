Geometry Utilities

Goal of Step 19: build the small kit of spatial primitives an engine leans on constantly — a ray, an axis-aligned bounding box, a plane, and the camera's view frustum — plus the missing Mat4 inverse they need. These are pure math (no GPU), so we can write them and pin them down with tests before any system consumes them.

They aren't four unrelated gadgets: they're one idea reused three ways. Box + frustum answers "can the camera see this?" (culling); ray + box answers "what's under the cursor?" (picking); box-vs-box is the cheap first pass of collision (physics). To make the step visibly pay off today, we also spend the new inverse on a real bug fix: the normal matrix, which finally lets a non-uniformly-scaled object be lit correctly.

1. First, the missing piece: a matrix inverse

A matrix transforms space; its inverse is the transform that exactly undoes it — inverse(M) · M = identity. If M moves a model into the world, inverse(M) brings a world point back into the model's own frame. We've gotten this far without one only because we always built the matrices we needed directly (the view matrix, for instance, is the camera's inverse assembled by hand in lookAt). Two jobs in this step finally force the general case:

screen pixel  ──inverse(proj·view)──▶  world-space ray      (picking)
model matrix  ──transpose(inverse(·))─▶  normal matrix        (correct normals under scale)

Our inverse is the textbook cofactor / adjugate method: build the matrix of signed minors, then divide by the determinant. If the determinant is ~0 the matrix is singular (some scale collapsed an axis, so information was lost and there's nothing to undo) — we return identity rather than emit NaNs, the same "degenerate input stays sane" stance normalize takes for the zero vector.

2. AABB — the axis-aligned bounding box

A bounding box is the smallest box that fully wraps a piece of geometry. Axis-aligned means its faces stay parallel to the world's X/Y/Z axes — the box never tilts, even if the object inside it does. That restriction is the entire point: because the faces are axis-parallel, "is this point inside?" and "do these two boxes overlap?" reduce to a few min/max comparisons per axis, with no rotation math at all. A tighter oriented box would cull a hair more, but the AABB's near-free tests are why culling, physics broadphase, and BVH trees are all built on it.

Aabb { Vec3 min;  Vec3 max; }         // two opposite corners define the whole box
contains(p)  =  min ≤ p ≤ max          (componentwise — 6 comparisons)

The subtle operation is transformed(M): turning a mesh's local box into a world box. You can't just transform min and max — once the box rotates it's no longer axis-aligned, so those two corners no longer describe it. Instead we transform all eight corners and fit a fresh AABB around them:

local box ──rotate 45°──▶ tilted box ──re-fit──▶ new, larger axis-aligned box
   the diagonal corner swings out, so the new bounds are WIDER than the original

That "build from nothing, fold points in" pattern needs an identity element, so Aabb::empty() returns a deliberately inverted box (min = +∞, max = −∞) that contains nothing; expand then grows it to swallow each corner. (A default {0,0,0}…{0,0,0} box would wrongly include the origin.)

3. Ray — a half-line through space

A ray is an origin plus a direction: the set of points origin + t·dir for t ≥ 0. The canonical use is picking — to find what the user clicked, you shoot the ray that starts at the camera and passes through the cursor's pixel out into the scene, then ask what it hits first. (Producing that ray is exactly the inverse(proj·view) "unprojection" from §1; wiring the cursor to it is a later step, but the ray/box test it needs lands here.)

Testing a ray against an AABB uses the elegant slab method. An AABB is the intersection of three "slabs" — the space between its two parallel faces on each axis. For each axis we compute the pair of t values where the ray crosses that slab's two faces, and keep the running overlap of the three [tenter, texit] intervals:

for each axis:  t0 = (min − origin)/dir,  t1 = (max − origin)/dir   (swap if dir < 0)
                tmin = max(tmin, t0)      tmax = min(tmax, t1)
hit  ⟺  tmin ≤ tmax          the ray is inside all three slabs at once

We start tmin at 0 so a box entirely behind the origin correctly reports a miss (the ray is a half-line, not an infinite line), and if the origin sits inside the box the entry distance clamps to 0. A neat bonus: for a ray running exactly parallel to an axis, dir = 0 makes t0/t1 come out as IEEE ±∞, and the min/max handle it for free — inside the slab it's ignored, outside it collapses to a miss.

4. Plane — and which side you're on

A plane is an infinite flat surface. We store it as a unit normal and a scalar d such that every point on the plane satisfies normal · p + d = 0. The value of that expression for an arbitrary point — its signed distance — is the workhorse: its sign says which side of the plane the point is on, and (for a unit normal) its magnitude is the actual distance to the plane.

signedDistance(p) = normal · p + d      > 0 : in front (normal's side)
                                        = 0 : on the plane
                                        < 0 : behind

That single test is the atom of visibility culling — because a frustum is just six planes, all facing inward.

5. Frustum — the shape the camera can see

Not everything in the world is on screen. The region a perspective camera actually captures is a frustum: a pyramid with its tip at the eye, sliced off by a near and a far plane — six planes in all (left, right, bottom, top, near, far). If we know these six planes we can, for each object, ask "is it inside?" and skip drawing the ones that aren't. That's frustum culling, and on a big scene it's the difference between drawing a thousand objects and drawing the fifty you can see.

                far plane
             ┌───────────────┐
              \             /
               \  visible  /        every point that survives the GPU's clip test
                \ region  /         lies on the INSIDE of all six planes
                 \       /
                  └─────┘  near plane
                     △ eye

Here's the beautiful part (due to Gribb & Hartmann): the six planes fall straight out of the view-projection matrix, no geometry required. Remember that a point's clip coordinate is just a row of that matrix dotted with the point. The GPU keeps a point only when its clip coordinates satisfy −w ≤ x ≤ w, −w ≤ y ≤ w, and the depth bounds. Rearrange each of those inequalities into the form "(some row combination) · p ≥ 0" and you've got a plane:

left   = row3 + row0        right  = row3 − row0
bottom = row3 + row1        top    = row3 − row1
near   = row2               far    = row3 − row2      ◀── note the near plane!

Watch the depth convention. Most textbooks write near = row3 + row2, because they assume OpenGL's clip range −w ≤ z ≤ w. Koi — like Vulkan, D3D and Metal, and matching our perspective() — maps depth to 0 ≤ z ≤ w instead. So the near-plane inequality is simply z ≥ 0, and the near plane is row2 alone. Use the OpenGL formula with our projection and objects mysteriously vanish at the near plane — a bug worth recognising on sight.

Testing a box against the frustum uses the positive-vertex trick. For each plane we only need to check the single AABB corner that reaches furthest in the plane's normal direction (the "positive vertex"). If even that corner is behind the plane, the whole box is, and we reject immediately:

for each of the 6 planes:
    p⁺ = the box corner furthest along this plane's normal
    if signedDistance(p⁺) < 0  →  box is fully outside  →  cull it
otherwise: visible (or straddling — which we conservatively keep)

The test is deliberately conservative: it may occasionally keep a box that only pokes into a frustum corner. That's fine — a culler must never wrongly discard something visible; a rare false "keep" just costs one extra draw.

6. Spending the inverse: the normal matrix (a real fix)

Now the visible payoff. Since Step 7 the vertex shader has carried surface normals into world space with the model matrix's upper-left 3×3, guarded by a comment that it "would need the inverse-transpose under non-uniform scale." We can finally lift that restriction.

The issue is that a normal is not an ordinary direction — it's a covector, defined by staying perpendicular to the surface. Apply a non-uniform scale (say 2× on X only) and an ordinary direction lying in the surface stretches with it, but a normal transformed the same way tilts off the surface — it's no longer perpendicular, and the lighting goes wrong:

scale ×2 on X:
   surface tangent  →   stretches, stays in the surface        (use  model)
   surface normal   →   naïve: tilts OFF the surface  ✗        (need  transpose(inverse(model)))
                        correct: the "normal matrix" keeps it ⟂ ✓

The transform that keeps a normal perpendicular is the inverse-transpose of the model's 3×3, the classic normal matrix. For an affine matrix, computing transpose(inverse(model)) on the full 4×4 and taking its mat3() gives exactly this — the translation coupling lands in the 4th column, outside the 3×3 block. So the CPU uploads one extra matrix, and the shader switches a single line:

// GpuRenderer, per draw                 // triangle.vert
struct VertexUniform {                    vWorldNormal  = mat3(normalMatrix) * inNormal;  // ← was mat3(model)
  Mat4 mvp;                               vWorldTangent = mat3(model)        * inTangent;  // stays on model
  Mat4 model;
  Mat4 normalMatrix;  // transpose(inverse(model))
};

Note the asymmetry: only the normal moves to the normal matrix. The tangent is a genuine surface direction, so it keeps riding model; the fragment shader then re-orthonormalises the pair (Gram–Schmidt) and rebuilds the bitangent, exactly as before. Get this backwards and normal-mapped surfaces light incorrectly under scale.

7. In the engine

  • src/math/Mat4.hpp — gains transpose and the general 4×4 inverse (free functions, like the rest of the matrix API).
  • src/math/Geometry.hpp — the new header: Ray, Aabb, Plane, Frustum, and the intersect(ray, box) test. Pure and header-only, like Vec / Quat.
  • shaders/triangle.vert and GpuRenderer.cpp — the normal-matrix wiring from §6.
  • tests/test_geometry.cpp — pins all of it down.

Because the demo scene uses only uniform scale, its lit frame is visually identical (under uniform scale the new normal differs only by a length that normalize() erases), while any non-uniformly-scaled object would now be lit correctly. It isn't quite byte-identical, though: like the quaternion swap in Step 18, a couple of hundred edge/highlight bytes differ in their last floating-point bits (max ≈ 23/255) — computing the normal matrix through a CPU inverse rounds a shade differently than the old mat3(model). That's FP noise, not a visible change. The tests carry the real proof:

inverse(M) · M              == identity            (affine AND perspective matrices)
box.transformed(rotZ 45°)    widens to ±√2 in x,y   (re-fit around the tilted corners)
ray(0,0,5 → −Z) vs unit box == hit at t = 4         (miss when the box is behind the origin)
frustum(proj·view) rejects a box behind the camera  (the z∈[0,1] near plane, working)
transpose(inverse(model))·N  stays ⟂ to model·T     (under non-uniform scale — the §6 fix)

8. Try it yourself

  1. Run the tests. cmake --build build then ./build/koi-tests — the new geometry suite proves every identity above.
  2. Confirm nothing moved. KOI_CAPTURE=out.bmp ./build/koi-demo renders the same frame as before; the normal-matrix change is invisible until an object is non-uniformly scaled.
  3. See the fix. Give a demo node a lopsided scale (e.g. node->transform().scale = {2, 1, 1}) and capture again: with the normal matrix its shading stays correct where the old mat3(model) path would have skewed the highlights.
