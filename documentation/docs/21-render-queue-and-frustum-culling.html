Render Queue & Frustum Culling

Goal of Step 20: stop drawing while we walk the scene. Until now the renderer recursed the node tree and issued a GPU draw at every node inline. This step splits that into two phases — traverse → flat list → submit — introducing a render queue. On top of it we add the first thing a queue makes possible: frustum culling, which drops objects the camera can't see before they cost a single draw call.

This is an architecture pivot, not just a feature. A flat draw list is the shared prerequisite for culling, sorting by material, instancing, transparency ordering, and deferred shading — none of which are expressible while drawing is tangled up with traversal. It also spends the Step 19 geometry layer (Aabb + Frustum) on its first real job.

1. The problem: draw-while-you-walk

Through Step 19 the main pass looked like this: a function recordNode received a node, and if that node had a mesh and a material it bound them and drew — then it called itself on each child. One recursive function did two completely different jobs at once: deciding what to draw (a tree walk) and submitting GPU commands (bind, push uniforms, draw). Fused together, they leave no seam to insert the questions every real renderer asks between "I found an object" and "I drew it":

recordNode(node):                        ← traversal AND submission, welded together
   if node has mesh+material: bind, draw
   for each child: recordNode(child)

   … but where do you put:
     "is it even on screen?"   (culling)
     "group draws by material" (sorting)
     "draw these 500 at once"  (instancing)
     "far-to-near for glass"   (transparency)
   ?  Nowhere — there is no list to reorder or filter.

2. The pivot: traverse → list → submit

The fix is to walk the tree once and flatten it into a plain array — the render queue — before drawing anything. Each entry is a RenderItem: a mesh, its material, the world matrix that places it, and (for this step) its world-space bounding box. Only after the list exists do we loop it and submit draws. Now the interesting operations all become passes over the list:

PHASE 1  traverse:  walk node tree → vector<RenderItem>   (buildRenderQueue)
PHASE 2  operate:   filter / sort / batch the list        (culling — this step)
PHASE 3  submit:    loop the survivors → bind + draw       (submitItem)

Separating the phases is the whole point: culling is a filter on the list, material sorting is a reorder of the list, instancing is a merge of adjacent list entries. This is why the roadmap calls the render queue the structural first move of the entire scaling track — every later performance feature operates here.

3. Giving every mesh a bounding box

To cull an object we need a cheap stand-in for its shape: its axis-aligned bounding box (Step 19). We compute each mesh's local (model-space) box once, at upload time, by folding every vertex position into an initially-empty box — and store it on the Mesh. (The mesh otherwise keeps only its GPU buffers; the CPU-side vertices vanish after upload, so this is the one moment they're still in hand.)

computeLocalBounds(vertices):  box = Aabb::empty();  for v: box.expand(v.position)

That's the box in the mesh's own frame. To test it against the camera we need it in world space, so as we build each queue entry we run the Step 19 Aabb::transformed(world) — which re-fits a fresh axis-aligned box around the eight transformed corners. Because a shared mesh (one cube, many nodes) has one local box but many world placements, the world box has to live on the item, not the mesh:

RenderItem {
  const Mesh*     mesh;         const Material* material;
  Mat4            world;        // where this instance sits
  Aabb            worldBounds;  // mesh.localBounds().transformed(world) — cached for culling
}

4. Culling to the frustum

A perspective camera sees only a frustum — the six-plane pyramid from Step 19. We extract those planes from the exact same projection · view the vertex shader uses (so "what we cull" and "what ends up on screen" can never disagree), then keep only the items whose world box intersects it. The test itself is Step 19's conservative Frustum::intersectsAabb; culling is just applying it to the list:

frustum = Frustum::fromViewProjection(proj · view)
visible = [ item for item in queue if frustum.intersectsAabb(item.worldBounds) ]
for item in visible: submitItem(item)      ← off-screen items never reach a draw call

An object entirely outside the frustum is dropped on the CPU — we skip the bind/push/draw calls entirely — which in turn saves all the GPU work those draws would have triggered (vertex shading, rasterisation, fragment shading). On the demo's handful of objects that's a rounding error; on a scene of thousands it's the difference between a slideshow and a game. The renderer logs a drawn / total count each second so you can watch the saving, and a runtime toggle (0) turns culling off for an A/B.

5. The shadow-pass trap

Here's the subtle bug this design is careful to avoid. We now render the scene twice: once from the light (the shadow depth map, Step 9) and once from the camera. It is tempting to cull both passes to the camera frustum. Don't.

camera can't see the tree behind it …           … but the tree's shadow falls INTO view
        ✗ cull it from the shadow pass  →  its shadow pops out of existence
        ✓ keep the shadow pass unculled  →  shadow stays correct

An object behind the camera can still cast a shadow onto something the camera can see. Cull its caster out of the shadow map and the shadow blinks in and out as you turn — a classic, maddening artifact. So the shadow pass draws the whole queue; only the camera pass is frustum-culled. (A future step can cull the shadow pass to the light's frustum instead — a different volume — but never to the camera's.) The render queue makes this easy to state precisely: same list, two different filters.

6. In the engine

  • src/renderer/RenderQueue.hpp / .cpp — the new layer: the RenderItem struct, computeLocalBounds, buildRenderQueue (the tree walk), and cullToFrustum (the filter). The two pure helpers are unit-testable with no GPU.
  • src/renderer/Mesh.hpp — each Mesh now carries its model-space Aabb, computed in GpuRenderer::createMesh.
  • src/renderer/GpuRenderer.cpp — renderSceneAndPost builds the queue once; the shadow pass loops it whole; recordScene culls then loops the survivors through the new submitItem (the old recursive recordNode / recordShadowNode are gone). setFrustumCullingEnabled is the runtime toggle.
  • tests/test_render_queue.cpp — pins down the bounds computation and the cull filter (in view kept, behind/beside culled, straddling kept, order preserved).

Because this is a pure restructuring of how we walk and submit — not a change to any math — the demo's captured frame is byte-for-byte identical to Step 19's. That exact-match is the proof the refactor is behaviour-preserving (the depth-first flatten keeps the original draw order, so even order-dependent blending would be unchanged). With every object on screen, culling removes nothing; turn the camera around and the log reads drew 0 / 9 (culled 9).

7. Try it yourself

  1. Run the tests. cmake --build build then ./build/koi-tests — the render-queue suite proves the bounds and the cull filter.
  2. Watch the counts. Run ./build/koi-demo and read the console: with the default view it logs drew 9 / 9 (culled 0). Fly (WASD + mouse) so objects leave the view and watch the drawn count fall.
  3. A/B the culler. Press 0 to toggle culling off, then on — what remains on screen is identical; only the drawn count (and, on a huge scene, the frame-rate) changes.
  4. Confirm nothing moved. KOI_CAPTURE=out.bmp ./build/koi-demo renders the same frame as Step 19, down to the byte.

8. What's next

➡️ With a flat list in hand, the scaling track opens up. The natural follow-ups all operate on the queue: sort it by pipeline / material to kill the per-draw rebinding cost (a known tax since Step 8); instance identical items into one draw call; and add transparency — which needs the queue, because blended objects must be drawn back-to-front, a sort you can only do once the draws are a reorderable list. The same Step 19 primitives also feed ray-cast picking and the physics broadphase. See the roadmap for the sequenced plan.
