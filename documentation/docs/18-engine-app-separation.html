Engine / App Separation

Goal of Step 17: stop letting the engine be the app. For sixteen steps the demo scene has lived inside the engine — Engine itself built the cubes, placed the lights, spun the hub, and read the keyboard. That was fine while there was only ever one thing to run, but it means the engine can't be reused: every scene would have to be edited into the engine's own source. This step draws a public API boundary. The engine keeps the machinery every app needs (window, renderer, the main loop); a new sample app under samples/ supplies all the content and behaviour, talking to the engine through a small interface — exactly the way a real game would.

No pixels change. This is a structural step: the same frame renders, but now nothing in src/ knows anything about the demo.

1. Two different jobs: engine vs. application

Up to now we've said "the engine" loosely, but there are really two jobs hiding in one word, and a reusable engine has to keep them apart:

  • the engine is the reusable machinery — the stuff every 3D program needs and no one wants to re-write: create a window, talk to the GPU, run a frame loop, measure time, route input. It should know nothing about your particular game.
  • the application is the specific content and behaviour — this scene, these lights, this camera, and how they react to time and keys. It's different for every program built on the engine.

The difference between a library and a program is exactly whether those two are separable. A program can tangle them; a library must not, because its whole point is that other programs supply the second half. That's why the project's roadmap calls this "the defining milestone for production apps": until the demo can be lifted out cleanly, the engine isn't something anyone else could build on.

2. The symptom: the engine hardcoded the demo

Before this step, Engine.cpp was full of the demo. It had a buildScene() that created the cubes and loaded the helmet, a setupLights() that placed the sun and the coloured lamps, a loop that spun the hub and orbited a light, and an event handler where keys 1–9 toggled bloom, IBL, the skybox and so on. And main.cpp was the app — it lived in src/, right beside the engine. Every one of those is demo-specific: a flight simulator built on Koi wants none of it. The engine had absorbed the app.

3. Inversion of control: "don't call us, we'll call you"

The fix is a classic idea with an intimidating name. Normally you think of your code calling library code: you call renderFrame(), you call loadTexture(). A frame loop turns that around. The loop has to live in the engine (only the engine knows how to acquire a swapchain image, pace to the display, and submit), but the contents of each iteration are the app's. So the engine must reach up into the app once per frame. That reversal — framework code calling into your code, rather than the other way round — is inversion of control, sometimes phrased as the Hollywood principle: "don't call us, we'll call you."

BEFORE  Engine::run()  ── calls ──▶  hardcoded buildScene / animate / toggles   (one fixed program)
AFTER   Engine::run(app)  ── calls back ──▶  app.onStart / onUpdate / onEvent    (any Application)

Concretely, the engine no longer holds a scene at all. Its loop asks the app what to do at each stage, through a handful of hooks the app implements. The engine is now genuinely content-free.

4. The Application interface

The boundary is one small abstract class, Application. An app subclasses it, holds its scene / camera / lights as members, and fills in the hooks; the engine's run() drives them:

Engine::run(app)
  │
  ├─ app.onStart(engine)          once — build the scene, load assets, place lights
  │
  └─ every frame, until quit:
        app.onEvent(engine, ev)   for each OS/input event (keys, mouse, close)
        app.onUpdate(engine, dt)  advance animation, read held keys (dt = seconds elapsed)
        app.frameView()           the engine asks "what do I draw?" → hands it to the renderer
  │
  └─ app.onShutdown()             once — release GPU resources (see §7)

The hook pattern is what keeps the loop plumbing — delta-time, event pumping, the headless KOI_CAPTURE path — inside the engine, written once, while the per-app parts vary. Two hooks have defaults (onEvent and onShutdown do nothing) so a trivial app only has to implement three. Notice each hook receives Engine&: that's how the app reaches back for a service it needs — most importantly engine.renderer(), which it uses in onStart to build meshes and textures, and engine.requestQuit(), which it calls to stop the loop (our demo does this on Esc).

5. FrameView: one bundle of "what to draw"

How does the app tell the engine what to render? Look at what the renderer actually needs for a frame: the camera's view matrix, the scene to walk, the eye position (for specular highlights), the lights, the clear colour, and the post-processing settings. Rather than pass those as six separate arguments — and pass them twice, because both the live path and the headless capture path need them — we bundle them into one value, FrameView:

struct FrameView {
    SDL_FColor             clearColor;   // background for uncovered pixels
    Mat4                   view;         // camera: world → view
    const Node*            root;         // the scene graph to draw (app-owned)
    Vec3                   cameraPos;    // eye in world space → specular
    std::span<const Light> lights;       // active lights (index 0 = the sun)
    PostSettings           post;         // tone-map / bloom / FXAA / vignette
};

The app returns one from frameView(); the engine hands it straight to renderFrame(fv) (live) or captureFrame(path, fv) (the BMP debug tool). Bundling matters for more than tidiness: because both paths consume the same struct, they can't drift apart, and a future field (a second camera, a time value) is added in one place instead of threaded through every call site. A FrameView owns nothing — its root points at the app's scene and lights spans the app's vector — so it's only ever used within the frame that produced it.

6. Who owns what now

The whole refactor is really one table — a clean split of responsibilities:

ENGINE  (koi_core, src/)          APPLICATION  (samples/demo/, a koi::Application)
  window + GPU renderer             the scene graph (Node tree)
  the main loop + delta-time        the fly camera
  event pump + OS-quit handling     the lights
  KOI_CAPTURE / KOI_MAX_FRAMES      the post-processing settings
  services: renderer(), quit()      animation + input reactions (the hooks)

The demo now lives in samples/demo/DemoApp.cpp as DemoApp : koi::Application. Its onStart holds the old buildScene/setupLights code (now calling engine.renderer() instead of a private member); its onUpdate spins the hub and orbits the light; its onEvent handles the number-key toggles. It sits under samples/, not src/ — that directory boundary is the point: koi_core compiles without ever seeing this file, and a real game would drop in beside it as a second Application.

7. A lifetime gotcha: free GPU resources in order

Splitting ownership introduces a subtle hazard worth understanding, because it's the kind of bug that only bites at shutdown. The app's meshes and material textures are GPU resources — they were allocated through the renderer's GPU device, and they must be freed through it too. But the device is owned by the engine. So if the app's scene is destroyed after the engine tears the device down, those frees dangle: a use-after-free on the way out.

That's exactly what onShutdown() is for. The engine promises to call it from run() while the renderer is still alive, and the app uses it to release its scene then and there:

Engine::run(app) {
    ... loop ...
    app.onShutdown();      // app frees its scene here — device STILL alive  ✔
}                          // control returns to main…
Engine::shutdown();        // …only now is the GPU device destroyed

DemoApp::onShutdown simply resets its scene-graph root, which cascades into freeing every mesh buffer and texture — safely, because the device outlives the call. This is the same "children die before the resource they borrow" discipline the engine has always used for teardown ordering; splitting the app out just made where that happens explicit.

8. The proof: identical pixels

A pure refactor should change structure, not output. We can hold ourselves to that literally. Our KOI_CAPTURE tool renders a single deterministic frame (the scene at time zero) to a BMP, so a capture from the new koi-demo can be compared byte-for-byte against one from the old build:

KOI_CAPTURE=after.bmp ./build/koi-demo
cmp before.bmp after.bmp   → identical  ✔   (same MD5)

An identical hash is the strongest evidence a refactor is behaviour-preserving: the demo does exactly what it did before — only its ownership moved. (This is also a preview of the next production step: once such captures are checked in as approved "golden images", a CI run can flag the day a change alters the render unintentionally.)

9. Try it yourself

  1. Run the sample. The build now produces koi-demo instead of koi-engine: cmake --build build, then ./build/koi-demo. Everything behaves as before — WASD to fly, the number keys to toggle effects and lights.
  2. See the boundary. Open Engine.cpp — it no longer mentions a single cube, light, or key. All of that moved to DemoApp.cpp. The engine got smaller.
  3. Write your own app. Subclass koi::Application, build a one-cube scene in onStart (via engine.renderer()), return a FrameView from frameView(), and run it from a five-line main() like samples/demo/main.cpp. You never touch src/ — which is the whole idea.
  4. Verify nothing changed. Capture a frame with KOI_CAPTURE=out.bmp ./build/koi-demo and diff it against a capture from before the step; the pixels are identical.

10. What's next

➡️ With the app lifted out, the highest-leverage follow-up is cross-platform CI with golden-image tests — running the suite plus KOI_CAPTURE comparisons on another backend (Linux/Vulkan) so "compiles clean, renders wrong" bugs can't hide. From there the learning thread resumes: quaternions (smooth rotations, the gateway to skeletal animation) and transparency (the first blended draws).
