glTF PBR Import

Goal of Step 16: stop hand-building materials and load a real one from a file. Since Step 12 the engine has had a full physically-based shader, and Step 13 gave it per-pixel texture, metallic-roughness, normal and AO maps — but every map so far was a synthetic BMP we generated with a Python script. This step imports the base-colour, metallic-roughness, normal, occlusion and emissive maps directly out of a glTF file, so we can load the industry-standard Damaged Helmet test asset and see all of that machinery run on real data.

Two things ride along because a real asset demands them: an emissive term (the helmet's panels glow), and sRGB handling for colour textures (without it, the imported albedo comes out visibly wrong).

1. What glTF actually is

glTF ("GL Transmission Format") is the runtime-friendly 3D format — often called "the JPEG of 3D". A glTF asset is three kinds of thing, stitched together by references:

  • a JSON document describing the scene: which meshes exist, how their vertex data is laid out, which materials they use, and how materials reference textures;
  • one or more binary buffers holding the raw bytes — vertex positions, normals, UVs, indices — as tightly packed arrays;
  • images (PNG/JPG) for the textures.

It comes in two packagings. A .gltf is the JSON as a text file, with the buffers and images sitting next to it as separate files. A .glb is the same content packed into a single binary file — the JSON plus one big blob with the buffers and images embedded. We use the .glb of the Damaged Helmet precisely because everything is in one file, which (as we'll see) makes loading its textures simpler.

We don't hand-write a glTF parser — the JSON-and-accessor bookkeeping has little to teach us about graphics, so since Step 9 we've used the single-header library cgltf for it (just as we use tinyobjloader for .obj). What's new here is that we finally read the material half of what cgltf parses, not just the geometry.

2. Image vs. texture vs. sampler

glTF is careful to separate three ideas that beginners often blur together — and the distinction is exactly the one Step 6 drew:

  • an image is the pixel data — a decoded PNG. Just colours.
  • a sampler is how to read an image: filtering (smooth vs. blocky) and wrap mode (repeat vs. clamp). No pixels of its own.
  • a texture pairs one image with one sampler — "read this picture this way".

A material then points at textures: base-colour texture, normal texture, and so on. So importing a material means: for each of its texture slots, find the referenced image, decode it to pixels, and upload it as one of our Texture objects. Our engine still reads every texture through a single shared sampler (we don't yet honour glTF's per-texture sampler settings — a small, deliberate simplification), so the interesting work is decoding the images.

3. cgltf hands us bytes; stb_image turns them into pixels

Here's a subtlety that trips people up: cgltf does not decode images. It parses the JSON and loads the binary buffers, but a texture's image is still a compressed PNG/JPG — cgltf just tells you where the compressed bytes are. Decoding those bytes into raw RGBA pixels is a separate job, and it's the one genuinely new dependency this step adds: stb_image — another single-header library, fetched and compiled exactly like cgltf. Our own loadTexture previously handled only BMP (via SDL, to avoid a dependency); now BMP still goes through SDL and everything else (PNG/JPG) goes through stb_image.

Where are the compressed bytes? Two cases, and the loader handles both:

EMBEDDED (.glb)   image → buffer_view → { buffer bytes, offset, size }   → stbi_load_from_memory
EXTERNAL (.gltf)  image → uri "textures/albedo.png"                       → resolve path → stbi_load

For the .glb helmet, every image is a buffer view — a byte range inside the file's embedded blob — so we decode straight from memory with no filesystem access. That's the whole reason we picked the .glb: it sidesteps path resolution entirely. (Base64 data: URIs are a third case we deliberately leave for later.)

4. sRGB vs. linear: the gamma trap

This is the concept most worth understanding, because getting it wrong is invisible until you load a real asset. Lighting maths must be done in linear light — where "twice the number" really means "twice the photons", so that adding two lights, or averaging texels for a mipmap, gives a physically correct result. But image files that store colour — a base-colour or emissive texture — are almost always saved in sRGB, a non-linear encoding that devotes more of its 256 levels to the darks, matching how human vision works. An sRGB texel value of 0.5 is not half-brightness; it's roughly 0.52.2 ≈ 0.22 of full light.

So a colour texture must be decoded from sRGB to linear before it's used in shading. If you skip that (as the engine did up to now — a documented Step 10 shortcut), every albedo comes out too bright and slightly washed-out, and the PBR maths is subtly off everywhere.

The clean fix is to let the GPU do it for free. GPUs have sRGB texture formats (R8G8B8A8_UNORM_SRGB): the stored bytes are identical, but every time a shader samples the texture the hardware converts sRGB → linear on the way out. So "fixing sRGB" is purely a choice of texture format — one flag, no pixel processing:

base-colour, emissive   → *_SRGB format   (COLOUR data → GPU un-gammas on sample)
metallic-roughness, normal, AO → plain UNORM (DATA, already linear — must NOT be touched)

The second line is the other half of the rule, and it matters just as much: metallic-roughness, normal and occlusion maps are data, not colour. A normal map's (0.5, 0.5, 1.0) encodes a direction; running it through an sRGB decode would corrupt the geometry. So only the two colour slots get the sRGB format; the data maps stay linear. The renderer's uploadToGpuTexture gained a single srgb parameter, and the loader passes true for exactly the base-colour and emissive images.

5. Emissive: surfaces that make their own light

Everything the shader did before was reflected light — surfaces lit by lamps or the environment. Emissive is different: it's light a surface emits on its own, like a glowing screen or the Damaged Helmet's lit vents. Because it isn't reflected, it doesn't participate in the lighting loop at all — it's simply added on top of the shaded result:

color = ambientIBL · ao + directLighting;         // everything reflected
color += texture(uEmissiveMap, uv).rgb · emissiveFactor;   // + light the surface emits

Two nice consequences fall out of that one added line. First, a material with no emissive map binds the neutral white 1×1 fallback and carries an emissiveFactor of zero, so the term adds nothing — every existing object is untouched, with no shader branching (the same fallback trick as Step 13). Second, a bright emissive pixel can cross the Step 10 bloom threshold, so it glows — which is the first time our bloom pass has had a genuine in-scene light source to feed on rather than just a bright sky. The emissiveFactor travels to the shader in a second vec4 of the per-material uniform, which grew from one vec4 to two — a bigger push, but still a single uniform buffer.

6. A model is geometry and a material

There's a small but meaningful design change in the loader. Through Step 9, loadModel returned just a Mesh — geometry only — and the engine attached a hand-written material. That can't express "use the material from the file", so loadModel now returns a LoadedModel: a mesh and an optional material (ModelLoader.hpp). glTF fills in the material; OBJ leaves it null (we still import OBJ geometry only). This is the natural shape — "a model is a shape plus an appearance" — and it's what lets the Damaged Helmet arrive fully dressed while the older sphere and torus keep the showcase materials the demo assigns them.

One ordering detail worth noting: the images are decoded straight out of cgltf's still-live buffers, so the material must be imported before cgltf_free releases the parsed data — the kind of lifetime care that RAII usually hides but that a C library makes explicit.

7. The orientation gotcha (and why it lives in the app)

Load the helmet and it arrives lying on its back. That's not a bug — it's a scope boundary. A glTF file is a whole scene graph: nodes with their own transforms, arranged in a hierarchy, and the Damaged Helmet's mesh sits under a node that rotates it upright (the model is authored Z-up; the node brings it to our Y-up world). We import only the raw primitive geometry and ignore the node transforms — a deliberate deferral — so the correction becomes our job: the demo gives the helmet its own rotationEuler in Engine.cpp to stand it up and turn it toward the camera. Importing the full node hierarchy is a future step.

Verify by looking, not by reasoning. Whether the helmet is upright, whether the albedo looks right, whether the vents glow — none of that shows up in a compile. As with every rendering step, we confirm with a captured frame: KOI_CAPTURE=out.bmp ./build/koi-engine renders one frame headlessly to a BMP. The first capture is how the orientation and the emissive strength get dialled in.

8. Try it yourself

  1. See it at all. The helmet downloads at CMake configure time. Build and run — KOI_CAPTURE=out.bmp ./build/koi-engine, then sips -s format png out.bmp --out out.png — and you'll see the imported base colour, metallic reflections of the sky (IBL on the file's metallic-roughness map), normal-mapped surface relief, and glowing emissive vents.
  2. Prove the sRGB fix matters. In ModelLoader.cpp, temporarily pass false for the base-colour image's srgb argument, rebuild, recapture: the helmet's paint turns pale and washed-out. Put it back.
  3. Turn the glow up. The emissive strength comes straight from the file. Multiply the imported emissiveFactor in loadGltfMaterial by, say, 4, and watch the vents bloom harder — the bright pixels sailing past the Step 10 threshold.
  4. Load your own. Point loadModel at any other .glb with embedded textures and it should come in fully materialled — the import isn't hard-coded to the helmet.
