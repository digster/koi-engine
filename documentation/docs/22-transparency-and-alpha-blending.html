Transparency & Alpha Blending

Goal of Step 21: make surfaces you can see through — glass, water, coloured film. Every object so far has been opaque: it completely hides whatever is behind it. A translucent object instead lets some of the background show through, so its final colour is a mix of the surface and everything behind it. Producing that mix is alpha blending, and getting it right forces two changes on the pipeline and one on the draw order.

This is the first feature to actually cash in the Step 20 render queue: blended objects must be drawn back-to-front, and you can only sort a flat list, never a tree walk.

1. Why order didn't matter — until now

Recall the depth buffer. For every pixel it stores the distance of the nearest surface drawn so far; each new fragment is kept only if it is closer, and when kept it writes its own depth back. That one mechanism makes opaque rendering order-independent: draw a thousand solid objects in any sequence you like and the depth test still leaves exactly the nearest one visible at every pixel. Draw order is irrelevant because a nearer surface completely replaces a farther one.

opaque fragment wins the depth test  →  it REPLACES the pixel   (dst is discarded)
    final = src                         ← the background contributes nothing

Transparency breaks that assumption at its root. A translucent fragment does not replace the pixel; it combines with it. So the background suddenly matters, and therefore so does the order in which things were drawn into that background.

2. The "over" operator: what blending computes

Give every fragment an alpha (α) — its opacity, from 0 (invisible) to 1 (solid). Compositing a translucent source fragment over whatever colour is already in the target (the destination) is the classic "over" operator:

out = src.rgb · src.α  +  dst.rgb · (1 − src.α)
        ▲ the surface's own    ▲ how much of the background
          colour, weighted        still shows through

At α = 1 this reduces to out = src (fully opaque, the background vanishes); at α = 0 it reduces to out = dst (the surface is invisible); at α = 0.4 you get 40% surface + 60% background — coloured glass. The GPU does this in fixed-function hardware at the very end of the pipeline: you don't write the formula in a shader, you configure it with blend factors. The source colour is multiplied by SRC_ALPHA, the destination by ONE_MINUS_SRC_ALPHA, and the two are ADDed — which is exactly the equation above.

3. Two reasons order now matters

(a) Blending is not commutative. "Red glass over blue glass" and "blue glass over red glass" are genuinely different images — whichever you composite last dominates. So overlapping translucent surfaces must be composited in a definite order: back-to-front, farthest first, so each nearer layer blends on top of the result so far. This is the painter's algorithm, the same reason a painter lays down the sky before the mountains before the foreground.

far pane, then near pane   ✓   near composites over far        (correct)
near pane, then far pane   ✗   far composites over near        (wrong — looks inside-out)

(b) The depth-write trick actively sabotages blending. If a translucent fragment wrote its depth like an opaque one, it would tell the depth test "I am the nearest surface here" — and a translucent surface behind it would then be depth-rejected and never drawn, so you'd never see it through the glass. The whole point of glass is that things behind it survive. So translucent draws must keep the depth test (they're still hidden by nearer opaque walls) but turn depth write off (they must not block each other). Correct front-to-back ordering then comes from our sort, not from the depth buffer.

4. Two pipelines, one shader

In SDL3's GPU API, blend state and the depth-write flag are baked into the pipeline object when it's created — they are not per-draw switches. So a renderer that draws both opaque and translucent geometry needs two pipelines. They share the exact same triangle.vert / triangle.frag and vertex layout; only two pieces of fixed-function state differ:

                        depth TEST   depth WRITE   blending
  opaque  pipeline          LESS          ON          off       (the Step 20 pipeline, unchanged)
  transparent pipeline      LESS          OFF         ON  (over operator, SRC_ALPHA / 1−SRC_ALPHA)

The fragment shader itself barely changes. It already computed a lit color; now it also emits an alpha instead of a hardcoded 1.0:

outColor = vec4(color, material.w * albedoSample.a)
                        ▲ opacity        ▲ the base-colour map's own alpha

material.w is the material's opacity (glTF calls it baseColorFactor.a), and it's multiplied by the albedo texture's alpha channel so a map can carve out see-through regions per texel. For an opaque material both are 1, so the alpha is 1.0 — and since the opaque pipeline has blending off, that alpha is ignored entirely. That's why adding this line changes nothing about how the existing scene renders: the opaque path is byte-for-byte the same, and only the new blend pipeline consumes the alpha.

5. The draw order, and a skybox subtlety

Putting it together, one render pass now runs in three stages:

1. OPAQUE      pipeline: depth-write ON   → fills the depth buffer, any order (depth sorts them)
2. SKYBOX      depth-write OFF, LEQUAL    → paints the background where nothing opaque was drawn
3. TRANSPARENT pipeline: depth-write OFF  → sorted far→near, each blends OVER what's already there

Why must the skybox come between the opaque and transparent stages, not last as it used to? The sky writes no depth and only fills pixels the opaque pass left at the far plane. If we drew the translucent panes before the sky, a pane sitting against empty background would blend over the clear colour — and then the sky, seeing that those pixels are still at far depth, would paint straight over the pane and erase it. Resolving the background first means a translucent surface correctly blends over the actual sky behind it. The render queue makes this reordering a one-line change: same list, three ordered sub-passes.

One nicety for free: because the scene is composited in the HDR target before tone-mapping, blending happens in linear light — which is the physically correct place to mix colours. Blending after gamma encoding (a common beginner bug) darkens the seams between layers; we sidestep it entirely.

6. Sorting the queue

The sort itself is a pure pass over the culled list. partitionByBlend walks the visible items once, dropping each into an opaque or a transparent bucket by its material's alphaMode. The opaque bucket keeps queue order (the depth buffer will sort it); the transparent bucket is sorted by distance from the camera, farthest first — using each item's world-space bounding-box centre (an Step 19 Aabb) as a cheap stand-in for its position.

partitionByBlend(visible, cameraPos, sort):
    for item in visible:
        (item.material.alphaMode == Blend ? transparent : opaque).push(item)
    if sort: stable_sort(transparent) by distance²(cameraPos, item.bounds.center) DESCENDING

Because it touches no GPU state, it is unit-tested headlessly: the split, the back-to-front order, the untouched queue order when sorting is off, and that it clears its outputs. A runtime toggle (T) turns the sort off so you can see the artifact it prevents.

7. Honest limitations

Per-object sorting is the simple transparency, and it is deliberately not perfect. Three cracks are worth naming out loud, because seeing them is how you understand why "real" transparency is a research topic:

  • Translucent objects cast solid shadows. The shadow pass draws the whole queue as depth-only and ignores materials, so a glass pane throws an opaque shadow. Fixing it (alpha-tested or stochastic shadow casters) is a later step.
  • The sort is per-object, by a single centre point. Two panes that interpenetrate, or one large translucent mesh that wraps around another, can still composite in the wrong order — there is no single "which is in front" for shapes that overlap in depth. This is exactly what motivates order-independent transparency (OIT).
  • Only hand-authored transparency. The glTF loader still imports every model as opaque; it doesn't yet read the file's alphaMode / baseColorFactor.a. The demo's glass is authored in code.

8. In the engine

  • src/scene/Material.hpp — a new AlphaMode enum plus alphaMode and opacity fields (added last, so existing initialisers are unchanged).
  • shaders/triangle.frag — samples the albedo as a full vec4 and outputs outColor.a = material.w · albedoSample.a.
  • src/renderer/RenderQueue.hpp / .cpp — partitionByBlend, the pure opaque/transparent split with the back-to-front sort.
  • src/renderer/GpuRenderer.cpp — createTrianglePipeline now builds a second, blend-enabled / depth-write-off pipeline from the same shaders; recordScene runs the opaque → skybox → transparent stages; bindFrameLighting re-establishes the shadow/IBL bindings after the pipeline switch. setTransparentSortEnabled is the T toggle.
  • samples/demo/DemoApp.cpp — two translucent glass panes at different depths in front of the scene.

9. Try it yourself

  1. Run the tests. cmake --build build then ./build/koi-tests — the render-queue suite now also proves the split and the back-to-front sort.
  2. Look through the glass. Run ./build/koi-demo: the whole scene is visible through the two coloured panes, tinted by each. Where the panes overlap, the nearer one blends on top.
  3. Break the sort. Press T to disable back-to-front sorting. When the panes overlap the compositing flips to the wrong order; press T again to restore it. That flip is the whole reason the sort exists.
  4. Capture a frame. KOI_CAPTURE=out.bmp ./build/koi-demo, then sips -s format png out.bmp --out out.png to view it.

10. What's next

➡️ The queue keeps paying off. Alpha-tested cutout (glTF MASK: keep the depth-write trick but discard fragments below an alpha threshold — foliage, fences) is the cheap half-way house between opaque and blended. Particle systems — camera-facing billboards with additive/alpha blends — are the natural next use of this blend pipeline. And sorting the opaque queue by material to cut per-draw rebinding, then instancing, are the remaining scaling wins the flat list unlocks. Farther out, order-independent transparency removes the per-object sort limitation entirely. See the roadmap for the sequenced plan.
