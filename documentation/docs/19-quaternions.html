Quaternions

Goal of Step 18: replace the way we store rotation. For every step so far a Transform has turned an object with three Euler angles — an X angle, a Y angle, and a Z angle, baked in a fixed order. That's easy to picture but has two well-known failures: it can gimbal-lock (silently lose an axis of freedom), and two orientations can't be blended smoothly — which is a wall the moment we want animation. This step introduces a hand-rolled Quat and switches Transform to store a single unit quaternion instead.

Nothing on screen changes — the demo renders the same frame. This is a representation step: the same rotations, stored in a form that doesn't lock up and can be interpolated. That second property is the real prize; it's the gateway to skeletal animation a few steps out.

1. Why Euler angles aren't enough

An Euler-angle rotation is three separate turns applied one after another. Our Transform composed them rotationZ · rotationY · rotationX — spin about X, then about Y, then about Z. Applied to a vertex, the matrices act right-to-left:

rotate = Rz(θz) · Ry(θy) · Rx(θx)          // three fixed-order turns
                       ▲
              the MIDDLE rotation decides where the OTHER two axes end up

Here's the trap. The three axes aren't independent — the middle turn (Y, here) physically moves the axis that the last turn is supposed to use. When the middle angle reaches 90°, it swings the X axis directly on top of the Z axis. Now your first and last knobs both rotate about the same physical direction: you've lost a whole degree of freedom, and no combination of the three angles can produce certain orientations near that pose. This collapse is gimbal lock — named after the physical rings ("gimbals") of a gyroscope that line up and jam in exactly the same way.

You've already seen us dodge it. The fly camera clamps pitch to just under ±90° precisely so its "up" axis never aligns with the world-up axis and flips. That clamp is a workaround, and it only works because a camera never needs to look straight up. An arbitrary object (a tumbling asteroid, an animated joint) has no such luxury.

The second problem is subtler but just as fatal for animation: you can't interpolate Euler angles cleanly. To animate an object from orientation A to orientation B you want to slide smoothly between them. Blending the angle triples component-by-component — lerp(Ax,Bx), lerp(Ay,By), lerp(Az,Bz) — does not trace a smooth turn: the object wobbles, speeds up and slows down, and can swing the long way around. We need a representation where "half-way between two orientations" is a meaningful, well-behaved thing. Quaternions are that representation.

2. Warm-up: rotating the 2D plane with a complex number

Quaternions look alien, so let's build intuition from something tamer that most people have met: complex numbers. A complex number a + bi is a point in the 2D plane. The magic fact is that multiplying by a unit complex number rotates the plane:

e^{iθ} = cos θ + i·sin θ          // a unit complex number = "rotate by θ"
(e^{iθ}) · (point)  →  that point, rotated by θ about the origin
(e^{iα}) · (e^{iβ}) = e^{i(α+β)}  // multiplying them ADDS the angles = composes rotations

So in 2D, rotations are unit-length complex numbers, and composing rotations is just multiplication. Quaternions are the 3D generalisation of exactly this idea — with one twist we'll meet in a moment. Hold onto the pattern: unit-length number ≙ rotation, multiplication ≙ composition.

3. What a quaternion is

A quaternion has one real part and three imaginary parts (Hamilton needed three to cover 3D):

q = w + x·i + y·j + z·k          // one real (w), three imaginary (x, y, z)

We only ever use unit quaternions — those with length 1 (x²+y²+z²+w²=1). The reason they matter is a beautiful correspondence: a unit quaternion encodes a rotation directly as an axis and an angle. To rotate by angle θ about a unit axis â = (âx, ây, âz):

      ┌ x ┐   ┌ sin(θ/2) · âx ┐
q  =  │ y │ = │ sin(θ/2) · ây │        w = cos(θ/2)
      │ z │   │ sin(θ/2) · âz │
      └ w ┘   └   cos(θ/2)    ┘        (we store x,y,z,w — vector part first, like glTF)

Why the half-angle? It looks arbitrary until you see how a quaternion rotates a vector. It doesn't multiply the vector once; it "sandwiches" it between the quaternion and its inverse, so the rotation gets applied twice — and each application contributes θ/2, which sum to the θ you wanted. That's the twist versus the 2D complex-number story.

4. Rotating a vector: the sandwich product

To rotate a 3D vector v by a unit quaternion q, treat v as a quaternion with zero real part, (0, v), and compute:

v' = q · (0, v) · q⁻¹            // "sandwich": q on the left, its inverse on the right

For a unit quaternion the inverse is just the conjugate q* = (−x, −y, −z, w) — flip the sign of the vector part — which is cheap. Multiplying that product out by hand gives a closed form with no quaternion algebra left in it (the Euler–Rodrigues formula), and that's what our Quat::rotate actually runs, because it costs a couple of dot/cross products instead of two full quaternion multiplies:

v' = 2·(u·v)·u  +  (s² − u·u)·v  +  2·s·(u × v)      where u = (x,y,z), s = w

The intuition to keep: q says "turn by θ about this axis," and the sandwich carries the vector along that turn. Rotating a normal or a position is the same operation.

5. Composing rotations: the Hamilton product

Just like the 2D complex numbers multiplied to add their angles, multiplying two quaternions composes their rotations. The rule (the Hamilton product) in scalar/vector form is:

q1 · q2  =  ( s1·s2 − v1·v2 ,   s1·v2 + s2·v1 + v1 × v2 )
             └── new w ──┘       └────── new (x,y,z) ──────┘
                                    scalar parts s, vector parts v

Two things to note. First, like matrices, it reads right-to-left: q1·q2 means "do q2's rotation first, then q1's." That cross-product term is why quaternion multiplication (like rotation itself) is not commutative — turning then rolling is not the same as rolling then turning. Second, this is exactly how Quat::fromEuler is built: it makes one quaternion per axis and multiplies them in the same qz · qy · qx order the old matrix path used — which is what let us swap the storage type without changing a single rendered frame (more in §8).

6. So why no gimbal lock?

Because a quaternion never decomposes the rotation into a fixed sequence of axis turns. It stores one axis and one angle for the whole rotation at once. There is no "middle" turn that can swing another axis onto a neighbour, so there is no pose where two controls collapse into one. Every unit quaternion is a valid, unambiguous orientation, and you can compose or interpolate them freely without ever hitting a singular configuration.

7. Slerp: smoothly blending two orientations

This is the property that makes quaternions non-negotiable for animation. Because every unit quaternion is a point on the surface of a 4-dimensional unit sphere, "blending two orientations" becomes a clean geometric idea: slide along the great-circle arc connecting the two points. That's slerp — spherical linear interpolation:

slerp(a, b, t)  walks the SHORTEST arc from a to b at CONSTANT angular speed
   t = 0   →  a
   t = 0.5 →  the true half-way orientation   (no wobble, no long way round)
   t = 1   →  b

An animation player stores a rotation per keyframe and calls slerp to fill in the frames between them; a camera can slerp from its current look to a target. Two subtleties our implementation handles, both worth understanding:

  • Double cover. A quaternion q and its negation −q represent the same rotation (θ about â equals −θ about −â). So before interpolating we check the dot product of a and b; if it's negative they're on "opposite sides" of the sphere, and we flip b so slerp takes the short way (≤ 180°) instead of the long way.
  • Near-parallel fallback. The slerp formula divides by sin(θ) between the two quaternions; when they're almost identical that's ≈ 0. There we fall back to a plain normalised lerp, which is indistinguishable at that tiny angle and doesn't divide by near-zero.

Our slerp is fully unit-tested this step (endpoints, the true midpoint, and the shortest-arc sign flip), even though the demo doesn't animate with it yet — it's the hook the animation step will grab.

8. In the engine: a drop-in swap for Transform

The new type lives in src/math/Quat.hpp — header-only and pure, exactly like Vec and Mat4. It offers fromAxisAngle, fromEuler, the Hamilton operator*, rotate, slerp, and a toMat4() that hands the renderer the same 4×4 model matrix it always consumed. Transform changed by one field:

// before                                  // after
Vec3 rotationEuler {0,0,0};                 Quat rotation {Quat::identity()};

Mat4 localMatrix() const {                  Mat4 localMatrix() const {
  return translation(position)                return translation(position)
       * rotationZ(e.z)*rotationY(e.y)             * rotation.toMat4()
       * rotationX(e.x) * scaling(scale);          * scaling(scale);
}                                           }

The T·R·S order and everything downstream are untouched — only the middle factor's source changed. Call sites read a touch nicer, too: standing the demo torus upright becomes Quat::fromEuler({radians(90),0,0}), and the spinning hub keeps a running angle and rebuilds its rotation each frame with Quat::fromAxisAngle({0,1,0}, angle).

9. Knowing when not to: the camera stays on yaw/pitch

A tempting overreach is to "quaternion-ify everything." We deliberately left the fly camera on its two Euler angles, and that's the right call — a genuine part of the lesson. An FPS/fly camera wants precisely two controls: turn left/right (yaw) and look up/down (pitch), with no roll, and pitch clamped so you can't tip over the top. Yaw + pitch expresses exactly those two degrees of freedom and structurally forbids roll. A free quaternion would add the ability to roll — a bug, not a feature, here — and buys nothing, since the clamp already sidesteps the only gimbal case a camera can reach. Quaternions shine for arbitrary orientation and interpolation; a camera's constrained aiming is a different job, and picking the representation that fits the job is the real skill.

10. The proof: same picture, greener foundation

Because Quat::fromEuler reproduces the old Rz·Ry·Rx rotation, a KOI_CAPTURE frame from the new build looks identical to one from before — the helmet stands upright, the cubes spin, the donut stands on edge, all unchanged. (Unlike the byte-identical refactor in Step 17, a handful of edge and highlight pixels differ in their last bits: rebuilding a rotation matrix via a quaternion rounds slightly differently than building it directly, which nudges a few antialiased silhouettes by a fraction of a shade. It's floating-point noise, not a visible change.) The new tests/test_quat.cpp pins the math down hard:

fromEuler(e).toMat4()  ==  rotationZ(e.z)·rotationY(e.y)·rotationX(e.x)   // behaviour-preserving
fromAxisAngle(Y,90°).rotate(+X)  ==  −Z                                   // matches rotationY
(a·b).toMat4()  ==  a.toMat4() · b.toMat4()                               // composition = matrix multiply
q · q⁻¹  ==  identity                                                     // inverse undoes it
slerp midpoint of 0°…90° about Y  ==  45° about Y, unit length            // shortest arc, constant speed

11. Try it yourself

  1. Run it. cmake --build build, then ./build/koi-demo — the scene behaves exactly as before; the change is entirely under the hood.
  2. Run the tests. ./build/koi-tests — the new quaternion suite proves the identities above (and that the swap kept every existing Transform/Node test green).
  3. Feel gimbal lock, then feel it gone. Set a Transform's rotation to Quat::fromEuler({radians(90),0,0}) and try to add a small extra yaw — with quaternions you can compose freely (extra · rotation) with no singular pose to trip over.
  4. Preview animation. In onUpdate, try node->transform().rotation = slerp(Quat::identity(), Quat::fromAxisAngle({0,1,0}, radians(90)), t) with t ramping 0→1 — a smooth, constant-speed quarter-turn. That's the mechanism a keyframe player will use.
