Meshes & a Scene Graph

Goal of Step 5: stop hardcoding geometry. Introduce a reusable mesh (geometry that lives in its own object, uploaded once and shared) and a scene graph (a tree of nodes with parent/child transforms) so objects can be composed, nested, and moved as a group.

To show it off, the scene is now a small hierarchy: a spinning hub cube with satellites that orbit it, one of which has its own little moon — plus a ground plane to sit on. Nothing here touches the shaders; it's all about how the CPU organises what to draw.

Through Step 4 the world was a flat, hardcoded list of cube positions, and the geometry itself was baked into the renderer. That doesn't scale: there's no way to add a different shape, and no way to say "this thing is attached to that thing." Step 5 fixes both, with two small abstractions.

1. From a list to a tree

In Step 4, drawing the scene looked like this — one shape, many placements, looped:

for (pos : hardcodedCubePositions)
    draw cube at pos

That's fine for confetti, but real scenes are hierarchical: a wheel is attached to a car, a moon orbits a planet, a hand is attached to an arm. When the car moves, its wheels should move with it for free, without us recomputing each wheel. The data structure that captures "attached to" is a tree — and a tree of placed objects is called a scene graph.

2. What a mesh is

A mesh is just a named bundle of geometry: a vertex buffer (the corners) plus an index buffer (which corners each triangle uses), living in GPU memory. We met both buffers in Step 2; the only new idea is giving them an owner — a Mesh object — so they're a thing the engine can hold, pass around, and reuse.

Reuse is the point. The scene has five cubes, but they're all the same shape. With a mesh, we upload the cube's 8 vertices to the GPU once and let every cube node point at that one mesh. Add a second shape (our ground plane) and it's simply a second mesh. The renderer no longer knows or cares what shapes exist — it just draws whatever meshes the scene hands it.

3. The Transform: position, rotation, scale

Where does a node sit, how is it turned, and how big is it? A Transform answers all three with three vectors — position, rotation (Euler angles, in radians), and scale — often abbreviated TRS. On its own that's just data; localMatrix() bakes it into the single 4×4 model matrix the GPU consumes (the "M" of the MVP chain from Step 3).

The subtle part is the order. We build the matrix as T · R · S, which — because a matrix is applied right-to-left to a point — transforms a vertex like this:

  localMatrix = translation · rotation · scaling

  applied to a vertex, right-to-left:
     translate( rotate( scale( vertex ) ) )

  1. SCALE first  — grow/shrink around the object's own origin
  2. ROTATE next  — spin the resized shape about its own origin
  3. TRANSLATE last — move the finished shape into place

This order is the intuitive one. Get it wrong and you get classic bugs: translate before rotate and the object swings around the world origin like it's on a string, instead of spinning in place. (A worked example pins this down in test_transform.cpp.)

4. Local space, world space, and the graph

Each node's Transform is expressed in its parent's coordinate space — its local space. To actually draw a node we need its world placement: where it ends up after stacking every ancestor's transform. The rule is one line, and it's the entire scene graph:

  world(node) = world(parent) · local(node)

Here's the hierarchy this step builds. The hub spins; everything under it inherits that spin:

  root  (a group — no shape, just a parent)
  ├── ground            (plane mesh, dropped to y = −2)
  └── hub               (cube; SPINS about Y)
      ├── satellite A   (cube, offset +X)      ─┐ orbit the hub
      ├── satellite C   (cube, offset +Z)      ─┘ because the hub turns
      └── spinner       (cube, offset −X; also SPINS)
          └── moon      (cube, offset from the spinner)

Because moon's world matrix folds in spinner's rotation and hub's rotation, the moon orbits the spinner while the spinner orbits the hub — two levels of motion, and we never wrote a line of orbit code. We only spin two nodes; the tree does the rest.

5. Two passes per frame

A scene graph splits each frame into two clean passes — the same split real engines use:

PassWhat it doesWhere
1. Update transformsWalk the tree top-down, computing and caching each node's world matrix from world(parent) · local.Node::updateWorldTransforms
2. RenderWalk the tree again; for each node that has a mesh, bind it and draw at the cached world matrix.GpuRenderer::recordScene

Keeping "figure out where everything is" separate from "draw it" is what makes parented motion automatic: the update pass resolves the whole hierarchy first, so by the time we draw, every node already knows its absolute place.

6. Shared meshes, group nodes & a lifetime rule

Two ownership ideas make the tree behave:

  • Children are owned (unique_ptr): a node solely owns its sub-tree, so destroying the root frees everything beneath it.
  • Meshes are shared (shared_ptr): one cube mesh is referenced by many nodes. A node may also have no mesh — a pure group (or "pivot") used only to position or spin its children. Our root is exactly that.

There's also a RAII lesson worth stating plainly. A Mesh frees its GPU buffers through the device — but the device is owned by the renderer, not the mesh. So:

Every mesh must be destroyed before the renderer destroys the device. The engine guarantees this by releasing the scene (and thus all meshes) before the renderer in Engine::shutdown(). Free them in the wrong order and you're asking a dead device to free a buffer — a crash, or worse, a silent one.

7. The code

The mesh — src/renderer/Mesh.{hpp,cpp}

A tiny RAII owner of a vertex buffer, an index buffer, and an index count. It holds a non-owning SDL_GPUDevice* purely to release those buffers in its destructor (hence the lifetime rule above). It's non-copyable and non-movable — copying would double-free the GPU handles.

Making meshes — GpuRenderer::createMesh & Primitives.cpp

The renderer is now a factory: createMesh(vertices, indices) reuses the same staging-buffer upload from Step 2 and wraps the result in a Mesh. Primitives.cpp defines the actual shapes — makeCubeMesh (the RGB color cube) and makePlaneMesh (the ground) — so the renderer itself stays geometry-agnostic.

The transform — src/scene/Transform.hpp

Header-only and SDL-free, like our other math. Just the three vectors and localMatrix() returning translation · rotationZ · rotationY · rotationX · scaling. (Step 5 also adds scaling() to Mat4.hpp, next to translation().)

The node — src/scene/Node.{hpp,cpp}

One element of the tree: a Transform, an optional shared Mesh, owned children, and a cached world matrix. The whole scene graph is this five-line recursion:

void Node::updateWorldTransforms(const Mat4& parentWorld) {
    world_ = parentWorld * transform_.localMatrix();   // this node's absolute placement
    for (auto& child : children_)
        child->updateWorldTransforms(world_);           // hand it down as the child's parent
}

Passing this node's freshly-computed world matrix down as each child's parentWorld is what resolves arbitrarily deep hierarchies in a single pass — no special case for grandchildren; they just work.

Drawing the tree — GpuRenderer::recordScene

Bind the one shared pipeline, combine projection · view once, then walk the tree. For each node that has a mesh, bind that mesh's buffers, push (proj·view) · world as the MVP uniform, and draw. Nodes with no mesh contribute only their transform. Both the live path and the headless capture share this one function, so they can never drift.

Building & animating — src/core/Engine.cpp

buildScene() assembles the hierarchy from the table above. The loop then nudges two rotations and resolves the tree before drawing:

hub_->transform().rotationEuler.y     += 0.6f * dt;   // satellites orbit the hub
spinner_->transform().rotationEuler.y += 1.5f * dt;   // the moon orbits the spinner
sceneRoot_->updateWorldTransforms();                  // propagate parent → child → grandchild
renderer_->renderFrame(clearColor, camera_.viewMatrix(), *sceneRoot_);

8. Try it yourself

  1. Re-parent something. Move satellite A to be a child of spinner instead of hub in Engine.cpp and watch its motion change completely — same node, different parent.
  2. Add a grandchild's grandchild. Hang another cube off moon. With no new logic, it inherits three levels of motion.
  3. Feel the TRS order. In Transform.hpp swap the multiply to put translation after scaling/rotation and see offset objects fling around the origin.
  4. New primitive. Add a makePyramidMesh to Primitives.cpp and drop one into the scene — the renderer needs no changes at all.
