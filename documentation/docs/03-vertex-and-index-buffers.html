Vertex & Index Buffers

Goal of Step 2: stop baking geometry into the shader. Put the corners of a shape into real GPU buffers and feed them to the pipeline — the way every real mesh gets drawn.

We graduate from a triangle to a quad (a rectangle made of two triangles), because the quad is the smallest shape that shows off the second new idea: an index buffer, which lets triangles share vertices instead of duplicating them.

In Step 1 the vertex shader invented its three positions out of gl_VertexIndex — there was no data to manage. That's a dead end: you can't load a model that way. The professional path is to hand the GPU a block of vertex data and a description of how to read it. That's this step.

1. Why a buffer at all? CPU memory vs GPU memory

Your C++ Vertex array lives in system RAM, owned by the CPU. The GPU is a separate processor with its own fast memory (VRAM), and — this is the key fact — it cannot reach into your program's RAM to read that array. A shader can only read data that already lives in a GPU resource.

A buffer is exactly that: a plain block of bytes that lives in GPU memory. To draw our own geometry we must first get the vertex data out of CPU RAM and into a GPU buffer. That copy is the heart of this step.

2. The two-step upload: staging buffers & the copy pass

Here's the wrinkle: the fast, GPU-only memory a vertex buffer wants to live in is usually not writable by the CPU at all. So you can't just create the buffer and memcpy into it. Modern GPU APIs solve this with a two-step move through a transfer buffer (also called a staging buffer):

  your Vertex[]            (1) create a CPU-visible
  in CPU RAM                   "transfer buffer"
      │                            │
      │  (2) memcpy                ▼
      └──────────────►  ┌────────────────────┐
                        │  transfer buffer    │  CPU can map & write this
                        └────────────────────┘
                                 │  (3) copy pass on the GPU
                                 ▼
                        ┌────────────────────┐
                        │  vertex buffer      │  GPU-only, fast; shaders read this
                        └────────────────────┘
  1. Create the real vertex buffer in GPU memory (the destination).
  2. Create a transfer buffer — CPU-visible staging memory. Map it (get a pointer), memcpy your vertices in, unmap it.
  3. Record a copy pass: a command telling the GPU to copy the bytes from the transfer buffer into the vertex buffer. Submit it.

You've seen this machinery before — in reverse. Our headless screenshot tool (captureFrame) uses a download transfer buffer to copy rendered pixels back from the GPU to the CPU. Uploading geometry is the mirror image: same parts, opposite direction. A copy pass is just the GPU's "move bytes between resources" mode, the sibling of the render pass.

Because this upload happens once at startup, we don't bother waiting on a fence: GPU command buffers run in the order they're submitted, so the per-frame draw (submitted later) is guaranteed to see the finished copy. We can even release the transfer buffer immediately — SDL defers the actual free until the GPU is done with it.

3. The vertex input layout: teaching the GPU to read your bytes

A buffer is just bytes; the GPU has no idea that byte 0 is a position and byte 8 is a color. When we build the pipeline we attach a vertex input layout that describes the bytes exactly. It has two halves:

  • A buffer description (per bound buffer): which slot it's bound to, its pitch (how many bytes from one vertex to the next — i.e. sizeof(Vertex)), and whether it advances once per vertex or once per instance.
  • One attribute per shader input: which location it feeds, which buffer slot it reads from, its format (e.g. two floats, three floats), and its byte offset within one vertex.
  one Vertex = 20 bytes (the "pitch")
  ┌────────────┬──────────────────┐
  │ position   │ color            │
  │ 2×float    │ 3×float          │
  │ offset 0   │ offset 8         │
  └────────────┴──────────────────┘
   →location 0   →location 1
    FLOAT2        FLOAT3

That location number is the single source of truth that must agree across four layers. Get one wrong and the GPU reads the wrong bytes — usually with no error, just a garbled image:

  GLSL                SPIR-V            MSL (Metal)            SDL pipeline
  layout(location=1)  →  decoration  →  [[attribute(1)]]   →   SDL_GPUVertexAttribute
  in vec3 inColor                                               .location = 1

4. Index buffers: why store a vertex once and reuse it

A quad is two triangles. The naïve way needs six vertices — but two of them are duplicates, because the triangles share an edge:

  Without indices (6 vertices, 2 duplicated):
     triangle A = v0, v1, v2
     triangle B = v2, v3, v0     ← v2 and v0 stored AGAIN

  With an index buffer (4 vertices + 6 indices):
     vertices = [v0, v1, v2, v3]          ← each corner stored ONCE
     indices  = [0,1,2,  2,3,0]           ← cheap 16-bit references

An index buffer holds small integers that point at entries in the vertex buffer. The GPU walks the indices, and for each one fetches that vertex. This decouples "how many corners exist" from "how many times they're used." For a quad it saves two vertices; for a real model — where each vertex is shared by many faces — it routinely cuts vertex data by more than half, and it's how every mesh format stores geometry. Our indices are 16-bit (Uint16), enough for 65,536 unique vertices; meshes larger than that switch to 32-bit indices.

So the per-frame draw changes from "draw N vertices" to "draw N indices": SDL_DrawGPUIndexedPrimitives instead of SDL_DrawGPUPrimitives.

5. The code

The vertex shader now has inputs — shaders/triangle.vert

The hardcoded arrays are gone. Each invocation receives one vertex's attributes through in variables, tagged with the locations from the layout above:

// triangle.vert  (GLSL 450)
layout(location = 0) in vec2 inPosition;   // from the vertex buffer, offset 0
layout(location = 1) in vec3 inColor;      // from the vertex buffer, offset 8
layout(location = 0) out vec3 vColor;      // varying → fragment shader
void main() {
    gl_Position = vec4(inPosition, 0.0, 1.0);
    vColor      = inColor;
}

The fragment shader is unchanged — it still just passes the interpolated color through. The shader stopped knowing the geometry; it now transforms whatever it's fed.

One vertex, described once — src/renderer/Vertex.hpp

A small struct defines the CPU-side layout. Its byte layout is the contract the pipeline's attributes describe, so we pin it with static_asserts (and matching unit tests in tests/test_vertex.cpp) — if it ever drifts, the build fails instead of the image silently corrupting:

struct Vertex {
    float position[2];  // location 0, FLOAT2, offset 0
    float color[3];     // location 1, FLOAT3, offset 8
};                      // sizeof == 20  → the vertex buffer "pitch"

The upload helper — src/renderer/GpuRenderer.cpp

uploadToGpuBuffer() performs the whole two-step move from §2 — create GPU buffer, create & map transfer buffer, memcpy, copy pass, submit — and returns the finished buffer. createGeometry() calls it twice: once for the 4-vertex vertex buffer, once for the 6-index index buffer. Defining the data is now ordinary C++:

constexpr std::array<Vertex, 4> vertices = {{
    {{-0.5f,-0.5f}, {1,0,0}},  // 0 bottom-left  red
    {{ 0.5f,-0.5f}, {0,1,0}},  // 1 bottom-right green
    {{ 0.5f, 0.5f}, {0,0,1}},  // 2 top-right    blue
    {{-0.5f, 0.5f}, {1,1,0}},  // 3 top-left     yellow
}};
constexpr std::array<Uint16, 6> indices = { 0,1,2,  2,3,0 };

Binding & drawing — recordQuad()

The pipeline now carries the vertex input layout. Each frame, inside the render pass, we bind the buffers and issue an indexed draw (one helper shared by the live frame and the screenshot path, so they can't drift):

SDL_BindGPUGraphicsPipeline(pass, trianglePipeline_);
SDL_BindGPUVertexBuffers(pass, 0, &vertexBinding, 1);
SDL_BindGPUIndexBuffer(pass, &indexBinding, SDL_GPU_INDEXELEMENTSIZE_16BIT);
SDL_DrawGPUIndexedPrimitives(pass, /*num_indices=*/6, 1, 0, 0, 0);

6. The result (and a teaser for Step 3)

You get a centered rectangle with a red / green / blue / yellow corner gradient. Look closely and you can see a faint diagonal seam from the bottom-left to the top-right corner: that's the shared edge between the two triangles, with each triangle interpolating its own corners — the index buffer made visible.

You'll also notice the "quad" is wider than it is tall, even though its NDC coordinates are a perfect square (−0.5 to +0.5 on both axes). That's because NDC is stretched to fill a non-square window (1280×720). Fixing that — making a square look square regardless of window shape — needs a projection matrix, which is exactly where Step 3 begins.

7. Try it yourself

  1. Recolor or move the corners. Edit the vertices array in GpuRenderer.cpp and rebuild — no shader recompile needed, because the data lives in C++ now, not the shader.
  2. Break the layout on purpose. Swap the two SDL_GPUVertexAttribute offsets (give position offset 8, color offset 0) and run with the GPU debug layer on — watch the colors and shape go wrong, a vivid lesson in why the layout must match the struct.
  3. Drop the index buffer. Try drawing the quad with six non-indexed vertices instead, to feel exactly what the index buffer was saving you.
  4. Capture it headlessly: KOI_CAPTURE=out.bmp ./build/koi-engine, then sips -s format png out.bmp --out out.png.
