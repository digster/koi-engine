Debug Draw

Goal of Step 22: make the engine's invisible spatial data visible. The last two steps added machinery you can't see — bounding boxes and a camera frustum (Step 19), then frustum culling (Step 20) that quietly skips off-screen draws. A wrong bound or a mis-built frustum is a silent bug: the numbers look fine and the picture looks almost right. Every engine solves this the same way — a debug draw API that renders throwaway lines on top of the scene: a box around a collider, an arrow along a normal, a wireframe of the camera's frustum.

It is also our first taste of drawing geometry that changes every frame — a transient GPU buffer, rebuilt from scratch each frame, in contrast to the static mesh buffers uploaded once at load.

1. You cannot debug space by reading numbers

Suppose an object mysteriously vanishes when you rotate the camera. Is its bounding box too small? Is the frustum plane facing the wrong way? Is the world matrix stale? You could print sixteen matrix floats and eight AABB corners to the console and try to picture them in your head — or you could draw the box and the frustum right there in the scene and see, instantly, that the box is half the size of the mesh. Spatial bugs are spatial; the fastest debugger for them is your eyes. That is the entire motivation for debug draw, and it's why it tends to arrive right after the first culling / bounds / physics code — the moment there's finally something invisible worth looking at.

2. A new primitive topology: the line list

Until now every draw has been made of triangles — the only thing a GPU can fill with pixels. The pipeline's primitive topology told it how to group vertices into shapes: TRIANGLELIST means "every three vertices are one triangle." But the GPU can also rasterize lines, and that's exactly what a wireframe is. We build a second pipeline whose topology is LINELIST: every two vertices are one independent line segment.

TRIANGLELIST   v0 v1 v2 | v3 v4 v5 | …     every 3 verts → a filled triangle
LINELIST       v0 v1    | v2 v3    | …     every 2 verts → a 1-pixel line

That is why the CPU side stores debug geometry as pairs of vertices. A box is not a special "box primitive" — it is just its 12 edges, i.e. 12 line segments = 24 vertices. A cross marker is 3 segments; a frustum is another 12. The shader that draws them is the simplest in the engine: transform the world-space endpoint by the camera's view-projection, output a flat colour. No lighting, no texture — a debug line is a colour you want to read, not shade.

3. Immediate mode and the transient buffer

A mesh is uploaded to the GPU once and drawn for years; its vertex buffer is static. Debug lines are the opposite: the set you want changes constantly — a box follows a moving object, the frustum is whatever the camera is doing this instant. So debug draw is immediate mode: each frame the app re-declares the lines it wants (clear(), then a burst of line() / box() / frustum() calls), and the renderer uploads that fresh vertex list into a transient GPU buffer and draws it in a single call.

static mesh      upload once at load        → draw every frame from the same buffer
debug lines      rebuild + re-upload each    → draw once, throw the buffer away
                 frame ("immediate mode")

Re-uploading raises a hazard question: what if the GPU is still reading last frame's buffer when the CPU wants to write this frame's? Our answer is the simplest correct one: build a brand-new buffer each frame and release the old one. SDL defers the actual free until the GPU has finished with it, so no two frames ever share a buffer and there is nothing to synchronise by hand. (A production engine would instead keep one growable buffer and rotate its backing storage — SDL's "cycle" mechanism — but that's an optimisation, not a concept, and debug draw is off in the common case anyway.)

4. Drawing an overlay: depth on, writes off

The debug lines are drawn inside the main scene pass, right after the opaque, skybox and transparent stages, into the same HDR target. Two deliberate choices about the depth buffer make the overlay behave:

depth TEST  ON    → a line behind a wall is correctly HIDDEN by it (occlusion reads right)
depth WRITE OFF   → the throwaway overlay never stamps the scene's own depth buffer

Testing but not writing is the same trick the transparent pass uses, for the same reason: the overlay should respect the scene's depth without corrupting it. Drawing into the HDR target (rather than as a separate pass after tone-mapping) has one big payoff — the lines show up in KOI_CAPTURE frames too, so headless verification sees exactly what the window does. The small cost: the tone-map compresses the line colours a touch and FXAA can soften a 1-pixel line, so we pick bright, saturated colours on the CPU side and they read fine.

5. The frustum trick: unproject the NDC cube

Drawing a box from an Aabb is easy — you already have its eight corners. But how do you draw the camera frustum, that truncated pyramid of visible space? Its corners aren't stored anywhere. The elegant answer reuses two earlier steps at once. In clip space, the frustum is just the unit cube: everything the camera can see maps into x,y ∈ [−1, 1] and (in Koi's Metal / Vulkan convention) z ∈ [0, 1]. And the matrix that maps clip space back to world space is the inverse of the view-projection — the Mat4 inverse we built in Step 19. So:

take the 8 corners of the NDC cube      (x,y ∈ {−1,+1},  z ∈ {0,1})
push each BACKWARD through inverse(viewProj)   → the frustum's 8 world-space corners
divide by w   (the perspective divide)  → corners spread into the truncated pyramid
connect the same 12 edges as a box

Feed it the camera's own projection · view and you draw a wireframe of exactly the volume Step 20's culler tests against. The demo freezes that matrix on a key press, so you can then fly the camera out of the frozen frustum and watch objects leave it — the culling you couldn't see in Step 20, finally on screen. (The far plane is a long way off, so the demo's capture freezes a short-range copy to keep the wireframe compact.)

6. Honest limitations

  • Colours shift slightly. Because the lines live in the HDR scene target, tone-mapping and FXAA touch them. That's an accepted trade for having them appear in captures; a crisp post-tone-map overlay pass is a later refinement (and would need to re-run for the capture path).
  • Occluded, not x-ray. The lines are depth-tested, so a box's back edges hide behind the object. Sometimes you'd rather see the whole wireframe through the geometry — an "x-ray" mode (depth-test off) is a natural toggle to add next.
  • No text yet. Debug draw shows shapes, not labels. Naming an axis or printing a value on screen needs text rendering, a separate upcoming step.

7. In the engine

  • src/renderer/DebugDraw.hpp / .cpp — the pure, GPU-free collector: DebugVertex plus line / box / frustum / ray / cross, all building a flat line-list vertex array. Unit-tested (including that frustum(identity) lands on the NDC cube with z ∈ [0,1]).
  • shaders/debug_line.vert / .frag — the minimal unlit line shaders (world position + flat colour in, solid colour out).
  • src/renderer/FrameView.hpp — a new non-owning debugLines span, the app's per-frame overlay carried across the engine/app boundary.
  • src/renderer/GpuRenderer.cpp — createDebugPipeline (LINELIST, depth-test on / write off), uploadDebugLines (the per-frame transient buffer), and recordDebug (one line-list draw at the end of recordScene). lastCameraViewProjection() exposes the culling matrix so the app can freeze it.
  • samples/demo/DemoApp.cpp — builds the overlay each frame (AABBs of every node, the frozen frustum, light-position crosses) and binds the G / F / L toggles; KOI_DEBUG_DRAW turns it all on for a headless capture.

8. Try it yourself

  1. Run the tests. cmake --build build then ./build/koi-tests — the debug-draw suite pins the box and (crucially) the frustum unprojection.
  2. Turn on the overlays. Run ./build/koi-demo and press G (bounding boxes), L (light icons), F (freeze + show the camera frustum). Notice a box's far edges are hidden behind its object — depth testing at work.
  3. See culling. With the frustum frozen (F), fly the camera backward: the amber wireframe stays put while the scene slides out of it — that's the volume the 0 frustum-culling toggle from Step 20 is testing against.
  4. Capture a frame. KOI_DEBUG_DRAW=1 KOI_CAPTURE=out.bmp ./build/koi-demo, then sips -s format png out.bmp --out out.png. (Without KOI_DEBUG_DRAW the capture is byte-for-byte identical to Step 21 — debug draw adds nothing when nothing is queued.)

9. What's next

➡️ Debug draw is the visual companion the rest of the tooling track builds on. A debug HUD (frame stats, toggles) and text rendering (labels for the shapes we can now draw) are the next overlays; ray-cast picking and a manipulation gizmo reuse these same lines to show what's selected; and when physics arrives, drawing contact points and collision volumes is how you'll debug it. An x-ray toggle (depth-test off) is a small immediate follow-up. See the roadmap for the sequenced plan.
