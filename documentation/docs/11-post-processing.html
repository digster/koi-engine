Post-processing

Goal of Step 10: stop drawing the scene straight to the screen. Instead render it into our own off-screen image, then run a chain of fullscreen passes over that image before showing it — the modern post-processing pipeline. We add the full stack: HDR tone-mapping (with exposure and gamma), bloom, a vignette, and FXAA anti-aliasing.

This is a larger step, but each effect is a small, self-contained pass — and you can toggle every one of them at runtime to see exactly what it does.

1. Why we need a separate pass at all

Until now the scene was drawn directly into the swapchain image — the very picture the OS puts on screen. That's fine as long as each pixel's final colour is decided the moment it's drawn. But a whole family of effects works on the finished image: blur it, adjust its brightness curve, smooth its edges. To do that a shader must read the pixels back — and you cannot read a texture you are at the same time drawing into (the GPU makes no promises about half-written pixels).

The fix is a two-phase structure. First render the scene into a texture we own (an off-screen render target, sometimes called a framebuffer). Then, in a second phase, sample that texture and write the processed result to the swapchain. Our engine already knew how to render off-screen — that's how the headless BMP captureFrame works — Step 10 just makes it the normal path for every frame.

2. The fullscreen triangle

A post pass has to run its fragment shader once per output pixel, so we need to draw geometry that covers the whole screen. The obvious choice is a quad (two triangles). The clever, standard choice is a single oversized triangle whose interior blankets the entire [-1,1] clip-space square; the parts hanging off-screen are simply clipped away.

   clip space (-1..1)          one big triangle covers it
   (-1, 3)                     ▷ no vertex buffer at all
      *                        ▷ 3 vertices from gl_VertexIndex
      | \                      ▷ no interior diagonal seam
      |   \                    SDL_DrawGPUPrimitives(pass, 3, ...)
      |     \
   (-1,-1)---( 3,-1)

Why a triangle and not a quad? A quad's two triangles meet along a diagonal that the GPU rasterizes twice (a subtle seam and wasted work); one triangle has no interior edge. Better still, it needs no vertex buffer: the vertex shader (fullscreen.vert) synthesizes the three corners from gl_VertexIndex alone, and the draw call is literally "draw 3 vertices."

3. HDR: giving tone-mapping something to do

Here's the key decision: the off-screen scene target is HDR (high dynamic range) — a 16-bit float per channel instead of the usual 8-bit 0..255. Why? A monitor can only show values in [0,1], but the light in a scene isn't so polite: a white surface in sun plus a sharp specular highlight can add up well past 1.0. An 8-bit target clips all of that to a flat 1.0 the instant it's written, throwing the information away.

By rendering into a float target we keep those bright values (2.0, 5.0, whatever) until the very end, when a tone-map curve decides how to fit them into the display range. Without HDR, tone-mapping and bloom would have nothing above 1.0 to work with — they'd be near no-ops. HDR is what makes the rest of this step meaningful.

4. Tone-mapping & exposure

Tone-mapping maps unbounded HDR light down into the [0,1] a screen can show. The naive way — just clamp() — flattens every bright area to identical white, destroying detail (a lit cloud and a lightbulb become the same blob). A tone-map curve instead gently compresses the highlights: bright gets brighter but never quite saturates, so shape and detail survive. We use the ACES filmic curve, the same family used in film and modern games.

Exposure is a single multiplier applied before the curve — exactly like a camera's exposure dial. Turn it up and dim details lift out of shadow (while highlights roll off gracefully instead of clipping); turn it down and a blown-out scene comes back. Try the [ and ] keys.

5. Gamma: the linear ↔ sRGB workflow

A subtle but important correctness point. All our lighting math — adding, multiplying, averaging colours — assumes linear light, where 0.5 really means "half the photons." But displays are sRGB: they expect a non-linear encoding where roughly 0.73 (not 0.5) shows as half-brightness, matching how human vision allocates precision to darks. If we hand linear values straight to an sRGB display, the midtones come out too dark.

So the final step of the composite pass gamma-encodes the colour (pow(color, 1/2.2)), converting our linear result into what the display expects. (For simplicity we still treat input textures as linear — a documented shortcut; a fuller pipeline would sRGB-decode them on the way in too.) Toggle tone-mapping off with key 1 and you'll see the difference the curve makes on top of this encode.

6. Bloom: extract, blur, add back

Bloom is the soft halo real cameras and eyes show around very bright things. We fake it in three stages, and this is where HDR pays off:

  1. Bright-pass (bright.frag): keep only pixels brighter than a threshold — the genuinely-glowing bits — and discard the rest. Because the scene is HDR, "bright" can mean truly > 1.0.
  2. Blur (blur.frag): smear those bright spots outward with a Gaussian. A true 2D blur is expensive (an N×N box of taps), so we use a separable one: blurring horizontally and then vertically gives the same result with only N+N taps. We run it at half resolution (bloom is blurry anyway) and ping-pong between two textures to widen the glow.
  3. Add back: the blurred glow is simply added on top of the scene in the composite pass, scaled by an intensity knob.

7. FXAA: cheap anti-aliasing, last

Aliasing is the stair-step "jaggies" on edges: each pixel is one point sample, so a slanted edge has to snap to the pixel grid. True anti-aliasing (MSAA) takes extra samples per pixel — accurate but costly. FXAA (Fast Approximate AA) instead runs as a cheap post step on the finished image: it finds edges by local brightness (luma) contrast and blends colour along them, softening the staircase. It can't invent detail that was never sampled, but it's nearly free.

Crucially, FXAA is the last pass, operating on the final gamma-encoded LDR image — because it reasons about perceived brightness, the same non-linear values your eye sees. Run it before tone-mapping and it would judge edges by the wrong numbers.

8. The whole render graph

Ordering isn't arbitrary: bloom reads HDR light (so it comes before tone-mapping), and FXAA reads the final sRGB image (so it comes last). Each arrow is one render pass.

  shadow pass ─▶ shadow map (depth)                    (Step 9, unchanged)
  scene pass  ─▶ sceneHDR  (RGBA16F) + depth           the scene, in HDR
  bright-pass ─▶ bloom  (½-res)   = threshold(sceneHDR)
  blur ×N     ─▶ bloom  (ping-pong H then V, separable Gaussian)
  composite   ─▶ ldr    = gamma( vignette( ACES( exposure·(sceneHDR + bloom) ) ) )
  FXAA        ─▶ swapchain  = edge-smooth(ldr)          → shown on screen

9. The code

The shaders — fullscreen.vert + four passes

One shared fullscreen vertex shader feeds bright.frag, blur.frag, composite.frag (exposure + ACES + vignette + gamma), and fxaa.frag. Each is registered in CMakeLists.txt and compiled GLSL → SPIR-V → MSL like the rest.

The pipeline — GpuRenderer

createPostResources builds the four fullscreen pipelines (no vertex input, no depth) and a linear-clamp sampler. ensureSceneTargets lazily creates and resizes the off-screen textures — the HDR scene target, the half-res bloom ping-pong pair, and the LDR target — mirroring how the depth buffer already tracks the window size. The scene pipeline's colour format changed from the swapchain's to the HDR one, so the scene now renders into sceneHdr_. renderSceneAndPost ties it together (shadow → scene → runPostChain) and is shared by both the live and headless-capture paths, so the captured BMP always matches the window.

The controls — Engine & PostProcess.hpp

PostSettings holds the knobs; the Engine toggles them from the keyboard and hands them to the renderer each frame, which pushes them into the shaders' uniforms. The pure helpers (half-resolution sizing, the luma weights, the ACES curve) live in the header so they can be unit-tested headlessly in test_post.cpp.

10. Try it yourself

  1. Toggle each effect. Keys 1 (tone-map), 2 (bloom), 3 (FXAA), 4 (vignette). Watching bloom or FXAA switch on and off is the fastest way to understand what each contributes.
  2. Ride the exposure. Hold ] to over-expose until the scene blooms and the highlights roll off; hold [ to bring it back. Notice how tone-mapping keeps the brights from clipping.
  3. Exaggerate the bloom. Lower bloomThreshold (e.g. to 0.4) and raise bloomIntensity in PostProcess.hpp — the whole scene hazes over, making the effect unmistakable.
  4. Break the order on purpose. Move the gamma encode out of composite.frag, or run FXAA before tone-mapping, and see why the pipeline is ordered the way it is.

11. What's next

Natural directions from here: multiple and point lights (with shadow cascades for large scenes), and pushing Material toward PBR — roughness/metalness and normal maps — for physically-based surfaces.
