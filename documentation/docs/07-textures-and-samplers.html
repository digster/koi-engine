Textures & Samplers

Goal of Step 6: paint surfaces with an image instead of a flat per-vertex color. Add texture coordinates (UVs) to each vertex, load an image and upload it into a GPU texture, create a sampler, bind them, and sample the texture in the fragment shader.

The roadmap pairs textures with lighting; we're doing them as two small steps. Lighting is Step 7 — this step is textures only.

Until now a surface's color came entirely from its vertices, smeared across each triangle by the rasterizer. That can't show fine detail — a brick wall, wood grain, a face — without an absurd number of vertices. A texture solves this: a separate image the shader looks up per fragment, so detail is independent of geometry.

1. Texture coordinates (UVs)

How does a fragment know where in the image to look? Each vertex carries a texture coordinate — a 2D point, conventionally called (u, v) — into the image. The image is addressed in a normalized 0…1 square regardless of its pixel size: (0,0) is one corner, (1,1) the opposite. The rasterizer interpolates the UV across the triangle just like it interpolates color, so every fragment gets its own spot to sample.

  image space (UV)                a triangle's vertices carry UVs;
  (0,0) ───────► u (1,0)          the rasterizer interpolates them
    │  ┌───────────┐              so each fragment samples its own
    │  │ . . . . . │              point in the image:
    ▼  │ . . X . . │
  v    │ . . . . . │                 fragment at UV (0.5, 0.5)
 (0,1) └───────────┘                 → the image's center texel

UVs live on the vertex, but they're a separate attribute from position: the same shape can be unwrapped onto an image however you like by choosing UVs. SDL3's GPU API (like Vulkan/Metal/D3D) puts (0,0) at the top-left of the image.

2. Why the cube grew from 8 to 24 vertices

Back in Step 2 our cube was 8 shared corners. That no longer works once each face needs its own UVs. A single corner is shared by three faces, and it wants a different UV on each (it's the top-left of one face, the bottom-right of another). A vertex can hold only one UV, so we split the cube into 6 faces × 4 corners = 24 vertices, each face independent.

This is a recurring pattern: an attribute that differs per face (UVs now, surface normals when lighting arrives in Step 7) forces unshared vertices. Sharing only works for attributes that are genuinely continuous at a corner (like position).

3. Getting an image onto the GPU

A GPU texture is an image stored in GPU memory in a layout the hardware samples efficiently. Like a vertex buffer, the fast memory it lives in isn't CPU-writable, so we use the very same two-step staging move from Step 2 — just in 2D:

  image file ──load──► CPU surface ──memcpy──► transfer buffer ──copy pass──► GPU texture
   (.bmp)            (SDL_Surface)            (CPU-visible)                  (sampled in shader)

Concretely: SDL_LoadBMP reads the file (a plain BMP, so no extra image library), SDL_ConvertSurface normalizes it to 32-bit RGBA matching our GPU texture format, then we map a transfer buffer, copy the pixels in, and record a copy pass with SDL_UploadToGPUTexture. If that sounds familiar, it's the exact mirror of captureFrame, which downloads a texture's pixels back to the CPU.

4. Samplers: how to read a texture

A texture is just the data. How to read it is a separate object — the sampler — because the same image is often read different ways. Two settings matter most:

SettingOptionsWhat it controls
FilterNEAREST / LINEARWhat happens between texels. NEAREST snaps to the closest texel (blocky, pixel-art); LINEAR blends the 4 nearest (smooth).
Address modeREPEAT / CLAMP / …What happens outside 0…1. REPEAT wraps (so UVs of 0…N tile the image N times); CLAMP holds the edge.

We create one shared sampler with LINEAR filtering and REPEAT addressing. REPEAT is what lets the ground plane tile the texture: its UVs run 0…6, so with REPEAT the checkerboard repeats 6×6 across the floor instead of being stretched once. The cubes keep UVs in 0…1, so each face shows exactly one copy.

5. Descriptor sets & binding

Shaders reach GPU resources through numbered slots grouped into descriptor sets. SDL3's GPU API fixes which set each kind of resource uses. We already met this: the vertex shader's MVP uniform buffer is at set = 1. For a fragment-stage sampled texture the set is 2:

// in triangle.frag
layout(set = 2, binding = 0) uniform sampler2D uTex;

We tell SDL the fragment shader uses one sampler (via loadShader's new sampler count), and each frame we bind the texture+sampler pair to slot 0 with SDL_BindGPUFragmentSamplers(pass, 0, …) — once, before drawing the scene, since every mesh shares the same texture this step.

6. Sampling in the shader

With the UV interpolated and the texture bound, the fragment shader is a one-liner. We look up the image at this fragment's UV and multiply by the interpolated vertex color, which now acts as a tint (a white tint shows the texture unchanged):

// triangle.frag
outColor = texture(uTex, vUV) * vec4(vColor, 1.0);

That's why the cubes look like a checkerboard washed with their old RGB-cube colors, while the gray-tinted floor shows the pattern more plainly.

7. The code

The texture — src/renderer/Texture.{hpp,cpp}

A small RAII owner of one SDL_GPUTexture, exactly like Mesh: it holds a non-owning device pointer to release the texture, so the same lifetime rule applies — textures must be freed before the renderer's device (the Engine resets the texture before the renderer in shutdown()).

Loading & uploading — GpuRenderer

loadTexture(path) does the load → convert → repack → upload, and uploadToGpuTexture is the 2D sibling of uploadToGpuBuffer. createSampler builds the one shared sampler. The image data itself lives in Primitives.cpp only as UVs; the picture comes from assets/uv_grid.bmp, copied next to the binary by CMake's koi_assets target (mirroring how shaders are built).

The vertex & shaders — Vertex.hpp, triangle.vert/.frag

Vertex gains a uv[2] (appended, so position/color keep their offsets); the pipeline gains a third vertex attribute; the vertex shader passes UV through; the fragment shader samples. The cube in Primitives.cpp is now 24 vertices with per-face UVs; the plane tiles its UVs 0…6.

Binding per frame — recordScene & Engine

The Engine loads the texture at startup and hands it to renderFrame / captureFrame (like it hands over the scene root); recordScene binds it once before walking the tree.

8. Try it yourself

  1. Switch to NEAREST filtering in createSampler and watch the checkerboard go crisp/blocky up close.
  2. Change the tiling. Bump the plane's UV range in Primitives.cpp from 6 to 1 (one giant tile) or 20 (tiny tiles) — REPEAT does the rest.
  3. CLAMP instead of REPEAT and set the plane UVs past 1.0: the edge texels stretch to the horizon instead of tiling.
  4. Drop your own image in. Replace assets/uv_grid.bmp with any 24-bit BMP and rebuild — koi_assets copies it automatically.

9. What's next

➡️ Step 7 — Phong lighting. Surfaces are still uniformly bright, ignoring where they face. Next we add per-vertex normals and a light, and shade each fragment with ambient + diffuse + specular (Phong) — which, like per-face UVs here, is what finally makes the 3D form read as solid. (Normals will split the cube to 24 vertices for the same reason UVs just did — already done.)
