A 3D Cube: MVP & Depth

Goal of Step 3: leave flat 2D behind. Draw a real, spinning 3D cube — which means writing our own vector/matrix math, transforming vertices with a Model-View-Projection (MVP) matrix passed through a uniform buffer, and adding a depth buffer so the cube's near faces correctly cover its far faces.

This is the step where the engine becomes 3D. Everything after — cameras, lighting, models — builds on these three ideas.

Until now our vertices were already in clip space: we typed coordinates in the −1…+1 box and they appeared on screen. That doesn't scale to 3D. Instead we author geometry in its own convenient space and let matrices carry it the rest of the way.

1. The spaces a vertex travels through

A vertex starts in model space (coordinates relative to the object's own origin — our cube spans −0.5…+0.5 around its center) and must end in clip space (what the GPU wants). It gets there by a chain of matrix multiplications, each moving it into the next space:

  model space ──[ Model ]──► world space ──[ View ]──► view/camera space ──[ Projection ]──► clip space
   (cube's own)            (placed in the      (relative to the           (ready for the GPU's
                            scene, rotated)     camera)                    perspective divide)
  • Model (M): places/rotates/scales the object in the world. Ours spins the cube and pushes it away from the camera.
  • View (V): moves the world so the camera sits at the origin looking down −Z. We leave V as the identity this step — a proper camera arrives in Step 4 — and fold the "back up so we can see the cube" distance into the Model translate. So for now MVP = Projection · Model.
  • Projection (P): applies perspective so farther things look smaller, and squashes the visible region into clip space.

We combine them into one matrix on the CPU (MVP = P · V · M) and the vertex shader does a single multiply: gl_Position = mvp * vec4(position, 1.0). Read the product right-to-left: M acts first, then V, then P.

2. Homogeneous coordinates: why vec4 and the "w"

You may wonder why a 3D position becomes a vec4 with a fourth component w = 1. That fourth coordinate is the trick that makes a single 4×4 matrix able to express translation and perspective, not just rotation/scale.

After the vertex shader, the GPU performs the perspective divide: it divides x, y, z by w. The projection matrix is built so that a vertex's distance from the camera ends up in w. Dividing by it is what makes distant geometry shrink toward the center — the very essence of perspective. A point uses w = 1; a pure direction would use w = 0 (so translation doesn't move it).

3. Column-major matrices (and why we don't transpose)

A matrix is 16 numbers; the only question is what order they're stored in. GLSL — and therefore the SPIR-V/MSL our shaders compile to — reads a mat4 as column-major: the first four floats are the first column. We store koi::Mat4 the same way, so we can copy our matrix straight into the shader's uniform with no flipping. Element (row r, col c) lives at index c*4 + r, which puts a translation in the last column:

       col0 col1 col2 col3
  row0 [ 0    4    8   12 ]      translation (tx, ty, tz)
  row1 [ 1    5    9   13 ]      lives at indices 12, 13, 14
  row2 [ 2    6   10   14 ]
  row3 [ 3    7   11   15 ]

4. Uniform buffers: per-draw constants

Vertex attributes (Step 2) vary per vertex. But the MVP matrix is the same for all 8 cube corners in a draw — it's a per-draw constant. The GPU has a dedicated, fast channel for exactly this: a uniform buffer.

SDL3's GPU API gives uniform buffers a streamlined path — you don't even allocate a buffer yourself; you "push" the bytes onto the command buffer and the shader reads them:

SDL_PushGPUVertexUniformData(cmd, /*slot=*/0, &mvp, sizeof(mvp));

Two details that must line up, or the shader reads nothing:

  • The shader declares the uniform in a specific descriptor set. SDL's convention for SPIR-V puts vertex-stage uniform buffers in set 1: layout(set = 1, binding = 0) uniform UBO { mat4 mvp; };.
  • We must tell SDL the shader uses one uniform buffer when we create it (num_uniform_buffers = 1), so it sets up the binding.

5. The depth buffer: who's in front?

Draw a cube's six faces and a problem appears: which fragments are visible? The naïve answer — "whatever was drawn last wins" (the painter's algorithm) — fails the moment shapes interpenetrate or you rotate them, because draw order has nothing to do with what's actually nearer the camera.

The fix is a depth buffer (a.k.a. z-buffer): a hidden texture, the same size as the screen, storing for each pixel the depth of the closest fragment drawn so far. With the depth test enabled, before the GPU writes a fragment's color it compares the fragment's depth against what's stored:

  new fragment depth  <  stored depth ?
        │                       │
       yes → it's closer:       no → something nearer is
       write its color AND      already there: discard the
       update the stored depth  new fragment

We clear the depth buffer to 1.0 (the far plane) at the start of each frame and use the comparison LESS, so smaller (nearer) depths win. SDL3's clip-space depth runs 0 (near) … 1 (far) — the Metal/Vulkan/D3D convention, unlike OpenGL's −1…1 — which is exactly what our perspective() matrix produces. With this on, the cube's geometry sorts itself out no matter the draw order or rotation.

6. The code

The math — src/math/Vec.hpp & Mat4.hpp

Hand-rolled, header-only, no GLM. Vec.hpp has Vec2/3/4 with dot/cross/length/normalize; Mat4.hpp has the column-major matrix with identity, multiply, translation, rotationX/Y/Z, and perspective (right-handed, depth 0…1). It's pure code, so it's covered by real unit tests in tests/test_math.cpp (e.g. that a 90° Z-rotation maps +X→+Y, and that perspective sends the near/far planes to z = 0/1).

The vertex shader — shaders/triangle.vert

layout(location = 0) in vec3 inPosition;            // now 3D
layout(location = 1) in vec3 inColor;
layout(set = 1, binding = 0) uniform UBO { mat4 mvp; };  // per-draw constant
layout(location = 0) out vec3 vColor;
void main() {
    gl_Position = mvp * vec4(inPosition, 1.0);      // model space → clip space
    vColor = inColor;
}

The cube — GpuRenderer.cpp createGeometry()

Eight corners (each colored by its position, so the cube is literally the RGB color cube) and 36 indices = 12 triangles, two per face — the index buffer from Step 2 now earning its keep by reusing 8 vertices instead of storing 36. The vertex's position attribute widened from FLOAT2 to FLOAT3 (see Vertex.hpp; sizeof(Vertex) is now 24).

Depth in the pipeline & the render pass

The pipeline turns on the depth test (compare_op = LESS, depth write on) and declares its depth-target format. Each frame we (re)create a depth texture matching the window size (ensureDepthTexture) and attach it to the render pass, cleared to 1.0. Then we build the MVP — a time-based angle for the live window, a fixed angle for the deterministic KOI_CAPTURE screenshot — push it, and draw.

A gotcha worth remembering: a zeroed SDL_GPUDepthStencilTargetInfo sets the load ops to LOAD (enum value 0), and SDL forbids cycle = true with a LOAD op. Because our depth format carries no stencil, we set the stencil load/store ops to DONT_CARE explicitly. Forgetting this caused a GPU hang (the debug layer asserted, the device never went idle) — a reminder that in these APIs, "all zeros" is a real configuration, not "unset".

7. The result

A solid, gently spinning RGB cube, viewed at an angle so three faces show. Two things to notice versus Step 2: it has real depth (near faces hide far ones, thanks to the z-buffer), and it's no longer stretched — the projection's aspect-ratio term makes a square cube look square in a wide window. That stretch we flagged at the end of Step 2 is now fixed.

8. Try it yourself

  1. Stop the spin. In buildMvp, pass a constant angle — the cube freezes; the depth test still keeps it correct.
  2. Break the depth test. Set enable_depth_test = false in the pipeline and watch faces paint over each other in the wrong order — the bug the depth buffer exists to prevent.
  3. Change the lens. Try a 90° vs 30° field of view in perspective() and see the cube balloon or flatten.
  4. Move the camera (sort of). Change the model translate's z from −3 to −2 and the cube comes closer; this is the camera distance we'll soon move into a real View matrix.
  5. Capture it: KOI_CAPTURE=out.bmp ./build/koi-engine, then sips -s format png out.bmp --out out.png.
