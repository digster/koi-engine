The First Triangle

Goal of Step 1: draw the iconic red/green/blue gradient triangle, using shaders we wrote ourselves.

The triangle is the "hello world" of 3D graphics. Getting one on screen means we've stood up the three pillars everything else needs: shaders, a graphics pipeline, and a shader-compilation toolchain. After this, drawing a spinning textured cube is "the same thing, with more data."

We keep the exact frame structure from Step 0 (command buffer → render pass → submit) and add two things inside the render pass: bind a pipeline and draw 3 vertices. The interesting part is everything that has to exist for those two calls to mean something.

1. What is a shader?

A shader is a small program that runs on the GPU, in parallel, on lots of data at once. Drawing a triangle uses two of them, run at two different stages of a fixed assembly line called the graphics pipeline:

  3 vertices
      │
      ▼
┌──────────────┐   runs once PER VERTEX — outputs each
│ Vertex shader│   vertex's position (and extra data like color)
└──────────────┘
      │
      ▼
┌──────────────┐   fixed-function: figures out which pixels the
│  Rasterizer  │   triangle covers, and INTERPOLATES the vertex
└──────────────┘   outputs across them
      │
      ▼
┌──────────────┐   runs once PER PIXEL ("fragment") — outputs
│Fragment shad.│   that pixel's final color
└──────────────┘
      │
      ▼
  the image

So the vertex shader answers "where does each corner go?" and the fragment shader answers "what color is each pixel?" The rasterizer in between is not programmable — it's the GPU's hardwired job to turn triangles into pixels.

2. Clip space: where do vertices "go"?

A vertex shader must output a position in clip space. After the GPU divides by the w component you get Normalized Device Coordinates (NDC): a fixed box where x and y both run from −1 to +1, with (0,0) at the center of the window. Anything outside that box is clipped away.

         +1  (top)
          │
   -1 ────┼──── +1     ← (0,0) is the center of the window
          │
         -1  (bottom)

Our triangle's three corners are written directly in NDC ((0, 0.5), (0.5, -0.5), (-0.5, -0.5)), so no math is needed yet. In Step 3, when we want real 3D, the vertex shader will multiply incoming positions by matrices to produce clip space — but the GPU's expectation (output clip space) never changes.

3. Why the triangle has a gradient

We give each corner a color — red, green, blue. But a triangle is mostly not its corners. What color is a pixel in the middle?

The vertex shader's extra outputs (here, a vec3 color) are called varyings. The rasterizer interpolates them: a pixel halfway between the red and green corners receives a 50/50 blend; a pixel in the dead center receives an even mix of all three. The fragment shader just reads whatever blended color it's handed. That automatic interpolation is the gradient — we never compute it ourselves.

4. The shader toolchain (the tricky part)

Here's the catch that makes graphics feel different from normal programming: GPUs don't run shader source code. They run compiled bytecode, and every backend wants a different bytecode dialect:

BackendShader format it consumes
VulkanSPIR-V
Metal (macOS)MSL (Metal Shading Language)
Direct3D 12DXIL

So we author once and compile to each target ahead of time. Koi Engine authors in GLSL (the language every graphics tutorial teaches) and runs a two-step compile at build time:

  triangle.vert (GLSL)
        │   glslc        (compile GLSL  →  SPIR-V)
        ▼
  triangle.vert.spv  ──────────────►  used directly on Vulkan
        │   spirv-cross  (translate SPIR-V  →  MSL)
        ▼
  triangle.vert.msl  ──────────────►  used on Metal (this Mac)
  • glslc (from Google's shaderc) compiles GLSL to SPIR-V, the Vulkan-standard intermediate format.
  • spirv-cross translates that SPIR-V into MSL so the Metal backend can use it.

One quirk to remember: spirv-cross renames the entry point from main to main0 when it emits MSL. So when we load the .msl variant we must tell SDL the entry point is main0, not main. Our loader handles this automatically.

(The original plan was to use SDL's shadercross tool for this, but its install is broken on this machine, and glslc + spirv-cross are already present and work cleanly — plus GLSL is friendlier to learn than HLSL.)

5. The graphics pipeline object

You can't just hand the GPU two shaders and say "go." Modern GPUs want all the state for a draw bundled into one immutable pipeline object, created up front. Ours bundles:

  • the vertex + fragment shaders,
  • the primitive type (TRIANGLELIST — every 3 vertices make one triangle),
  • the color target format (must match the swapchain we draw into), and
  • fixed-function state we leave at defaults for now (no blending, no face culling, no depth test).

Why immutable? Because switching pipelines is the expensive operation, the GPU wants them pre-validated. We build ours once at startup and just bind it each frame — cheap.

6. The code

The shaders — shaders/triangle.vert & .frag

The vertex shader bakes the 3 positions and colors into arrays and selects one with gl_VertexIndex (the 0/1/2 index of the vertex being processed — no vertex buffer needed yet):

// triangle.vert  (GLSL 450)
layout(location = 0) out vec3 vColor;          // varying → fragment shader
vec2 positions[3] = vec2[](
    vec2( 0.0,  0.5), vec2( 0.5, -0.5), vec2(-0.5, -0.5));
vec3 colors[3]    = vec3[](
    vec3(1,0,0),      vec3(0,1,0),       vec3(0,0,1));
void main() {
    gl_Position = vec4(positions[gl_VertexIndex], 0.0, 1.0);
    vColor      = colors[gl_VertexIndex];
}
// triangle.frag  (GLSL 450)
layout(location = 0) in  vec3 vColor;          // interpolated by the rasterizer
layout(location = 0) out vec4 outColor;        // the pixel's final color
void main() { outColor = vec4(vColor, 1.0); }

The location = 0 on the vertex output and the fragment input must match — that's the wire connecting the two stages.

The build step — CMakeLists.txt

A koi_add_shader() function registers the two compile commands per shader (glslc then spirv-cross), and a koi_shaders target builds them into build/shaders/ — the folder the executable looks in at runtime. The app depends on it, so the shaders are always up to date.

Loading the right variant — src/renderer/Shader.cpp

selectShaderVariant() asks the device which format it supports and returns the matching file extension + entry point (.spv/main on Vulkan, .msl/main0 on Metal). loadShader() then reads shaders/<name><ext> from beside the executable and creates the SDL_GPUShader. (That variant logic is a pure function, so it's covered by unit tests in tests/test_shader.cpp.)

Pipeline + draw — src/renderer/GpuRenderer.cpp

At startup we load both shaders and build the pipeline (then release the shader objects — the pipeline keeps what it needs). Each frame, inside the render pass, the draw is just two calls:

SDL_BindGPUGraphicsPipeline(pass, trianglePipeline_);
SDL_DrawGPUPrimitives(pass, /*vertices=*/3, /*instances=*/1, 0, 0);

No vertex buffer is bound — the 3 vertices are conjured by the vertex shader from gl_VertexIndex. The GPU runs the vertex shader 3 times, the rasterizer fills the triangle, and the fragment shader colors each pixel.

7. Try it yourself

  1. Recolor the corners. Change the colors[3] array in triangle.vert and rebuild — the gradient follows. (The build recompiles shaders automatically.)
  2. Move a corner. Edit a positions[3] entry; push one past ±1 and watch it get clipped at the window edge.
  3. Inspect the compiled output. Open build/shaders/triangle.vert.msl to see the Metal code spirv-cross generated from your GLSL — note the main0 entry point.
