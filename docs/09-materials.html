Materials

Goal of Step 8: let objects look different. Introduce a Material — a bundle of surface appearance (its texture + how shiny it is) — that each scene node references, so the renderer can give every object its own look instead of one texture and one shininess for the whole scene.

The demo: a matte checker floor, glossy polka-dot cubes, and an extra-shiny hub (same texture as the other cubes, just a tighter highlight).

Through Step 7 the renderer bound one texture for the whole frame and the fragment shader hardcoded the shininess. So everything necessarily looked the same. That's the limitation this step removes — and in doing so it draws a clean line the engine has been missing.

1. Shape, placement, appearance

A drawn object is really three independent things, and the engine now has a type for each:

QuestionAnswered bySince
What is its shape?Mesh (vertex + index buffers)Step 5
Where is it / how oriented & scaled? (its placement)Transform on a NodeStep 5
What does it look like? (its appearance)MaterialStep 8

These are orthogonal: the same cube mesh can appear at many placements with different appearances. Keeping them separate is exactly why one cube Mesh, uploaded once, serves five differently-placed, differently-textured cubes.

2. What a material is

A Material is just the appearance bundle:

struct Material {
    shared_ptr<Texture> texture;       // the image sampled as the base color
    float shininess    = 32.0f;        // specular exponent (higher = tighter highlight)
    float specStrength = 0.4f;         // how bright the highlight is
};

It only describes appearance — it owns no GPU object of its own beyond a shared reference to a Texture. (The sampler — how to read a texture — stays renderer-owned and shared, since it rarely needs to vary.) Materials are themselves shared: every cube can point at one Material, just as they point at one Mesh. A node now draws only if it has both a mesh (shape) and a material (appearance).

3. Binding per draw, not per frame

With one global texture, the renderer bound it once before walking the scene. Now different nodes use different textures, so the texture bind moves inside the per-node loop — bound just before each draw, right alongside that node's vertex/index buffers (which were already rebound per node). Same rhythm, one more resource:

  recordScene:  bind pipeline · push light (once) · walk tree
    recordNode (per drawn node):
        bind material.texture (+ sampler)   ← NEW: was once-per-frame
        push material params  (shininess…)  ← NEW
        bind mesh.vertex / index buffers
        push { mvp, model }
        draw

There's a real cost here: more bindings and uniform pushes than the single global texture. Engines claw it back later by sorting draws so objects sharing a material are drawn together (fewer switches). We keep it simple and correct for now — optimization is a later concern.

4. Frame-constant vs per-object uniforms

The fragment shader now reads two uniform buffers, and the split is meaningful: one changes per frame, the other per object.

UniformSet / bindingPushedHolds
Lightset=3, binding=0once per frame (recordScene)light dir/color/ambient + camera
Materialset=3, binding=1per draw (recordNode)shininess, specular strength

SDL maps push slot N → binding N, so we push the light to slot 0 and each material to slot 1 (telling SDL the fragment shader has two uniform buffers when we load it). The shader just reads material.x / .y where it used to use the hardcoded constants.

5. The demo: appearance varies — and so do its parameters

Three materials make the point on two axes:

  • Floor — the cool checker texture, matte (low shininess).
  • Satellites & moon — a different, warm polka-dot texture, glossy.
  • Hub — the same polka-dot texture as the satellites, but extra shiny (a tighter, brighter highlight).

The floor-vs-cubes difference shows different textures; the hub-vs-satellites difference shows that a material's parameters vary independently of its texture. Both fall out of the same Material abstraction.

6. The code

The material — src/scene/Material.hpp

A header-only struct (texture + two floats). It lives in scene/ next to Node and forward-declares Texture (it only stores a shared_ptr to one), so it stays lightweight.

The node — src/scene/Node.{hpp,cpp}

Gains a shared_ptr<Material> beside its mesh, with setMaterial/material() and a (mesh, material) constructor — mirroring the existing mesh plumbing (verified in test_node.cpp).

Binding it — GpuRenderer & triangle.frag

recordNode binds the node's material texture and pushes its params per draw; recordScene no longer binds a global texture (it still pushes the light). The fragment shader adds the material UBO at set=3, binding=1.

Building them — Engine & CMake

buildScene loads the two textures, makes the three materials, and assigns them to nodes — the scene owns them, so the old global texture_ is gone. CMake's koi_assets now copies every assets/*.bmp next to the binary.

7. Try it yourself

  1. Make a mirror. Crank a material's shininess to 256 and specStrength up — watch the highlight shrink to a sharp glint.
  2. Swap textures. Give the floor the dots and the cubes the checker in buildScene — one line each.
  3. Add a third texture. Drop a new .bmp in assets/, re-run CMake (the glob picks it up), and build a material from it.
  4. Find the cost. Give all five cubes the same material and note that the renderer still rebinds it per draw — the motivation for sorting by material later.

8. What's next

➡️ Models & beyond. The engine now cleanly separates shape, placement, and appearance — exactly what a real model file (glTF/OBJ) carries. Next steps load such models (real geometry instead of hand-built cubes), then add shadows (objects occluding the light) and post-processing. A material is also the natural home for the richer surface data those bring — normal maps, roughness, metalness — on the road toward physically based rendering.
