Instancing & Draw-call Sorting

Goal of Step 24: draw the same picture with fewer, cheaper draw calls. Until now the renderer has issued one draw call per object, and each one re-binds that object's textures and mesh and pushes a fresh transform — a per-draw tax we've paid since the first cube. This step spends the render queue the way its own header promised: sort the draw list so identical work is adjacent (killing redundant state changes), then instance — collapse a run of identical objects into a single draw call that stamps the mesh once per copy.

It's the engine's first real performance step, and it changes how every object is drawn: the per-object transform moves out of a uniform and into a per-instance buffer the GPU walks itself.

1. What a draw call actually costs

A "draw call" (SDL_DrawGPUIndexedPrimitives) looks cheap — one function call — but each one makes the driver do real work: validate the bound state, translate it into GPU commands, and flush any state changes you made first (a new pipeline, new textures, a new vertex buffer, a pushed uniform). Ten thousand objects means ten thousand of these, and the CPU — not the GPU — becomes the bottleneck, spending all its time describing work instead of the GPU doing it. The two classic cures are complementary:

SORT    reorder draws so identical state is adjacent   → stop re-binding what didn't change
INSTANCE collapse identical draws into ONE call        → one "draw N copies" instead of N draws

Both only became possible at Step 20, when drawing was split from traversal into a flat list of RenderItems. You can reorder a list; you can't reorder a recursive tree walk mid-drawing.

2. Sorting: make identical work adjacent

Opaque objects can be drawn in any order — the depth buffer resolves what's visible regardless. That freedom is the opening: if we sort the opaque list by what state each draw needs, all the draws sharing a texture set and a mesh end up next to each other, and the renderer can bind that state once for the whole run instead of re-binding it for every object. Koi sorts the opaque items by a batch key of (material, mesh) — a stable sort, so equal-key items keep their order and the frame stays deterministic.

One caveat: transparent objects can not be reordered. Alpha blending is order-dependent (Step 21's back-to-front rule), so the translucent panes keep their far→near order and are drawn one at a time. Only the opaque pass is batched.

3. Instancing: one draw, many copies

Sorting sets up the bigger win. Instanced rendering tells the GPU: "draw this mesh N times." A single call renders every copy, and inside the vertex shader a built-in counter, gl_InstanceIndex, runs 0, 1, 2, … so each copy can look up its own data. The mesh's vertices and indices are read once; only the per-copy data changes.

Where does that per-copy data come from? A second vertex buffer — but bound at a different input rate. A normal vertex attribute advances once per vertex; an instance-rate attribute advances once per instance. So the GPU fetches a fresh transform each time it moves to the next copy, while re-reading the same mesh vertices underneath.

buffer slot 0  (per-VERTEX)     position, colour, uv, normal, tangent   → advances every vertex
buffer slot 1  (per-INSTANCE)   this copy's model + normal matrix        → advances every instance

  DrawIndexed(indexCount, num_instances = N)     ← N copies, ONE call

4. Why the vertex shader had to change

Here is the conceptual shift. Before this step, each draw pushed a uniform holding that one object's matrices (mvp, model, normalMatrix). A uniform is constant for the whole draw — which is fine when a draw is one object, but useless when a single draw now covers many objects that each need a different transform. So the transform moves out of the uniform and into the per-instance buffer, and the uniform shrinks to the one thing every copy really does share — the camera:

before (per draw):  UBO { mat4 mvp; mat4 model; mat4 normalMatrix; }   // one object's transforms
after  (Step 24):   UBO { mat4 viewProj; }                             // the camera, shared by all
                    in  mat4 inModel;         // per-instance attribute (locations 5–8)
                    in  mat4 inNormalMatrix;  // per-instance attribute (locations 9–12)
                    → gl_Position = viewProj * inModel * position;      // MVP formed in-shader

A mat4 attribute occupies four consecutive attribute locations (one per column), so inModel takes 5–8 and inNormalMatrix takes 9–12, right after the mesh's five per-vertex attributes (0–4). The normal matrix is still transpose(inverse(model)), precomputed on the CPU per instance — inverting a matrix in the shader would be wasteful, and it's needed because the demo's satellites are non-uniformly scaled.

5. Two passes, two batch keys

The engine draws the scene twice: once for the shadow map (depth from the light's view) and once for the lit image. Both get instanced — but they batch on different keys, and the reason is instructive:

COLOUR pass   key = (material, mesh)   a batch must share BOTH textures AND geometry
SHADOW pass   key = (mesh)             depth writes no colour — MATERIAL is irrelevant, so mesh alone

The shadow pass ignoring material means it forms bigger batches: in the demo it draws every cube (whatever its material) in one call and every plane in another. It's also the pass that draws the whole scene — it is deliberately never camera-culled, because an object behind the camera can still cast a shadow into view.

6. In the engine

The logic that decides what to draw is separated from the code that submits it — the same spirit as the render queue. A new prepare phase runs before any render pass begins:

  • Pure helpers in RenderQueue: sortByMeshMaterial (one stable sort by pointer key — mesh primary, material secondary) and coalesceBatches, which walks the sorted list and emits one DrawBatch {mesh, material, first, count} per run of equal keys. No GPU state → unit-tested headlessly (tests/test_render_queue.cpp).
  • GpuRenderer::buildFrameBatches ties it together each frame. It sorts the whole queue once by (mesh, material) and derives both passes from that single sorted list — the shadow pass coalesces it by mesh, the colour pass culls it (survivors stay grouped) then coalesces by (mesh, material) — packing every visible transform into a transient instance buffer. It runs up front, so the instance-buffer upload (which submits on its own command buffer) is finished before the passes draw.
  • The record passes then just bind + draw: for each opaque DrawBatch, bind the material + mesh once and issue DrawIndexed(indexCount, num_instances = batch.count), pointing the instance buffer at that batch's slice via a byte offset.

The instance buffers are transient — rebuilt and re-uploaded every frame (the transforms change as things animate), released-and-recreated exactly like the debug and HUD buffers, so no two frames ever share one and nothing needs hand-syncing.

7. Seeing the win

The HUD from the last step earns its keep here: it shows a live Draws N (M items) readout — the gap is exactly what batching saved. In the demo the four cubes that share a mesh and a material collapse into one instanced draw, so the colour pass reports 8 draws for 11 items; the shadow pass, batching by mesh alone, folds all five cubes and all three planes together and reports 5 draws for 11 items. Small numbers here — but the mechanism is the one that scales to thousands of copies for the cost of one call. And because the image is unchanged, this is a pure win: a Step-24 frame is visually identical to Step 23 (differing only in a handful of pixels, from the harmless reordering of opaque draws changing float rounding by a bit).

8. Honest limitations

This is CPU-side batching, and it's the first rung. Each object is still an item the CPU visits — we've cut the draw calls, not the per-object bookkeeping. The next rungs on the scaling track push the work onto the GPU: GPU-driven culling (a compute shader decides visibility), an indirect draw that reads its parameters from a GPU buffer, and eventually a render graph to manage the growing web of passes. Transparent objects also still draw one-per-call (order-dependent blending forbids batching them) — a limitation order-independent transparency would lift.

9. Try it yourself

  1. Watch the counter. cmake --build build && ./build/koi-demo, press H for the HUD, and read the Draws / items line. Fly around: as objects leave the view they're culled, so both numbers drop together.
  2. Read the batching log. The renderer logs Batching: N items → M color draws … shadow N → K draws — the same numbers, plus the shadow pass's bigger mesh-only batches.
  3. Confirm parity. KOI_CAPTURE=out.bmp ./build/koi-demo renders the same scene as before this step — instancing changes how it's drawn, not what is drawn.

10. What's next

➡️ With sorting + instancing in place, the scaling track opens up: GPU-driven culling and indirect draws (move the per-object work off the CPU entirely), deferred / clustered shading (decouple lighting cost from geometry so many lights stay cheap), and a render graph to tame the pass wiring. The instance buffer built here is also the natural home for a future particle system's per-billboard data. See the roadmap for the sequenced plan.
