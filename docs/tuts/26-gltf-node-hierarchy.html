glTF Node Hierarchy & Multi-material Meshes

Goal of Step 25: import a whole glTF scene, not just one mesh. Back at Step 16 we loaded a single object — the Damaged Helmet — by reading the file's first mesh's first primitive and its one material. Real glTF files are richer: a tree of nodes, meshes made of several primitives that each carry their own material, and textures shared across those materials. This step teaches the loader to honour all of that and hand back a ready-to-render scene-graph subtree.

The lovely part: the engine already has everything to draw such a subtree. This is almost entirely a loader step — plus one small piece of new math.

1. A glTF file is a scene, not a mesh

It's tempting to picture a model file as "a bag of triangles." glTF is more structured than that — it's a small scene description. The pieces, from the top down:

scene     a list of root NODES to show
node      a placement (transform) + optional mesh + child nodes   ← a TREE
mesh      a name + a list of PRIMITIVES                           ← NOT a single blob
primitive one draw: vertex attributes + indices + ONE material
material  base colour / metallic-roughness / normal / … maps + factors
texture   an image + a sampler (how to filter/wrap it)
image     the actual PNG/JPG bytes (embedded, or an external file)

Step 16 walked straight to meshes[0].primitives[0] and ignored the rest — fine for a one-piece asset, but it drops the node tree, every extra mesh, and every extra primitive on the floor. To load a real scene we have to visit all of it.

2. A mesh is a bag of primitives — and why

The surprising word above is primitive. In glTF a "mesh" is not one lump of geometry; it's a list of primitives, and the reason is materials. A single draw call can only bind one material at a time (one set of textures, one set of factors). So an object that needs several materials — say a character whose body, eyes, and armour look different — is split into one primitive per material. A car body and its glass windows are two primitives of the same mesh.

Our koi::Node draws exactly one (mesh, material) pair. That's not a limitation to work around — it maps the concept perfectly. A glTF mesh with N primitives simply becomes N child nodes, one per primitive, each with its own geometry and material, all sharing the parent node's placement:

glTF:  node "car"  ─ mesh ─ primitive 0 (body   material)
                            └ primitive 1 (glass  material)

koi:   Node "car" (group, holds the transform)
         ├─ Node (body mesh,  body material)     ← drawable
         └─ Node (glass mesh, glass material)    ← drawable

3. The node graph is our scene graph

A glTF node carries a local transform and a list of child nodes — a child's placement is expressed relative to its parent. That is exactly the scene graph we built in Step 5, where world(node) = world(parent) · local(node). So importing the hierarchy is a direct translation: each glTF node becomes a koi::Node carrying its transform, its primitives hang off it as child draw-nodes, and its glTF children recurse the same way.

Because the mapping is one-to-one, nothing downstream changes. Once the subtree exists, the app's per-frame updateWorldTransforms() pass propagates matrices down it, and Step 20's buildRenderQueue flattens it into draw items — the same code that already draws the hand-built demo scene. Even Step 24's instancing "just works": each imported sub-mesh is its own unique geometry, so it renders as a batch of one, sorted next to anything that shares its material.

4. Two ways to write a transform: TRS or a matrix

How does a glTF node store its placement? In one of two forms, and a loader must handle both:

TRS     translation[3] + rotation[4] (a quaternion, x,y,z,w) + scale[3]
MATRIX  a single 4×4 matrix (column-major) that already bakes T, R, S together

The TRS form drops straight into our Transform. This is a quiet payoff from the quaternion step: glTF stores a rotation as (x, y, z, w) — vector part first, scalar last — which is exactly our Quat's memory order, so the four numbers copy over with no shuffling.

The matrix form is the interesting one. Our Transform stores separate T, R, and S, but the file handed us a single baked matrix — so we must run the bake backwards, recovering the three parts. This is called decomposition:

TRANSLATION  the last column (the matrix's 12th–14th floats)
SCALE        the LENGTH of each of the three basis columns
             (a pure rotation's columns are unit-length, so any extra length IS the scale)
ROTATION     what's left after dividing each column by its scale — a clean
             orthonormal 3×3 → turned back into a quaternion

That last arrow needs a new tool: a quaternion from a rotation matrix, the inverse of the toMat4() we wrote in Step 18. It lives in Quat::fromRotationMatrix, and it uses the standard trace-based method — the naïve formula loses all precision near a 180° turn, so we solve for whichever component is provably largest first. The whole decomposition is Transform::fromMatrix.

Why is decomposition always safe? In general a 4×4 matrix could contain shear (a skew that no translation/rotation/scale can reproduce). The glTF spec forbids shear in node matrices — they must be a clean translate·rotate·scale — so the decomposition above is always exact. One subtlety: a mirrored node (a negative scale that flips handedness) can't be a rotation with all-positive scale, so we detect the flipped determinant and fold the sign into one scale axis, leaving a proper rotation the quaternion can represent. Both the round-trip and the mirror case are pinned down in tests/test_transform.cpp.

5. Uploading each texture only once

A scene's materials share textures — a tiling brick image might be the base colour of a dozen wall materials. Our texture loader (Step 6) decodes and uploads a fresh GPU texture every time it's called, so importing naïvely would upload that brick image a dozen times, wasting decode time and — worse — VRAM, holding a dozen identical copies.

The fix is a small cache that lives for the duration of one import: a map from a glTF image to the Texture we already made for it. The first material to reference an image decodes and uploads it; every later material that names the same image gets the shared handle back. It's the seed of the asset manager the roadmap plans — here, kept deliberately tiny and scoped to a single file.

6. In the engine

All of this is additive — the simple single-mesh loadModel stays for assets like sphere.glb, and a new entry point handles whole scenes. It all lives in src/renderer/ModelLoader.cpp:

  • loadModelHierarchy(renderer, path) — the new public entry point. Parses the glTF, picks the default scene, and returns a std::unique_ptr<Node> subtree (wrapped in one group node) plus a per-import texture cache. The caller attaches it with sceneRoot_->addChild(...).
  • buildGltfNode(...) — the recursion. One glTF node → a group koi::Node (its local transform from gltfNodeTransform), one child draw-node per primitive, then recurse into child nodes.
  • loadPrimitiveMesh(...) — the per-primitive geometry read (positions, normals, UVs, tangents, indices), extracted from the old single-mesh path so both loaders share one, well-tested code path. Missing normals/tangents are still filled in as before.
  • gltfNodeTransform / Transform::fromMatrix / Quat::fromRotationMatrix — the transform plumbing from §4. The pure-math pieces live in src/scene/Transform.hpp and src/math/Quat.hpp, unit-tested with no GPU.

Notice what didn't change: Node, the render queue, the batching, the shaders. The engine was already built to draw a deep tree of many-material objects — this step just fills that tree from a file.

7. Honest limitations

This step imports the structure faithfully, but a few glTF features are deliberately left for later — most of them only start to matter for the big stress scene (Sponza) that comes next:

  • Samplers / wrap modes aren't read yet. Every texture uses the one renderer-owned sampler, so a texture authored to tile (repeat) won't tile correctly until we import per-texture wrap modes — the first thing Sponza will force.
  • data-URI images (base64 pixels inlined in the JSON) are still skipped, as in Step 16; we handle embedded-in-the-.glb and external-file images.
  • Alpha mode from the material isn't imported — everything comes in opaque even though the material supports blending.
  • Every primitive becomes its own node. For a huge scene that's a lot of small nodes — finer culling granularity (good) but more bookkeeping (a cost a future asset manager will address).

8. Try it yourself

  1. Generate the test model. uv run tools/gen_multimesh.py writes assets/multimesh.glb — a tiny scene built to exercise every path at once: a root pivot (a 25° turn) parents a two-material post (one mesh, two primitives) and a cap whose placement is a raw matrix, and the two post materials share one checker texture.
  2. Run it. cmake --build build && ./build/koi-demo — the model appears front-left. The loader logs Loaded model hierarchy '…multimesh.glb' (1 root node(s), 1 unique texture(s)): two textured materials, one upload — the cache at work.
  3. Capture it. KOI_CAPTURE=out.bmp ./build/koi-demo then sips -s format png out.bmp --out out.png. You'll see the stacked, tinted, checker-textured post with its wide flat cap correctly placed above it — proof the multi-primitive fan-out and the matrix decomposition both landed.

9. What's next

➡️ With hierarchies and multi-material meshes in hand, the loader is ready for a real environment: Sponza — which will pull in glTF samplers/wrap modes (so its tiled textures read right) and put the culling and lighting under genuine stress. From there the learning thread branches to alpha-tested cutout (foliage), cascaded shadow maps, and — building on the quaternions and this importer — skeletal animation.
