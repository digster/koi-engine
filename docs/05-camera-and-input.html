A Fly Camera & Input

Goal of Step 4: move through the scene. Add a first-person fly camera driven by the keyboard and mouse, a view matrix built by lookAt, and a delta-time clock so motion is smooth and frame-rate independent.

To make the movement obvious there are now several cubes scattered around — a plain hardcoded list, not yet a scene graph (that's Step 5).

In Step 3 the View in "Model-View-Projection" was the identity matrix — the camera was stuck at the origin. This step makes V real, and wires up the input to control it.

1. There is no camera — only a view matrix

This is the idea that makes cameras click: the GPU has no concept of a camera. The vertex shader only knows clip space. So we don't "move a camera" — we move the entire world by the opposite of the camera's transform. Slide the camera two metres right, and equivalently every object shifts two metres left. Turn the camera, and the world counter-rotates around you.

The view matrix is exactly that "inverse of the camera" transform. It takes world-space coordinates and re-expresses them relative to the camera. We build it with lookAt(eye, center, up):

  lookAt builds an orthonormal camera basis and the matching shift:
     forward f = normalize(center - eye)   // where the camera points
     right   s = normalize(cross(f, up))   // its right
     up      u = cross(s, f)               // its true up

  view = | s.x  s.y  s.z  -dot(s,eye) |    rotation rows re-express the
         | u.x  u.y  u.z  -dot(u,eye) |    world in the camera's axes;
         |-f.x -f.y -f.z   dot(f,eye) |    the last column shifts by -eye
         |  0    0    0       1       |

Our full transform per vertex is now Projection · View · Model. (Read right to left: model places the object in the world, view moves the world relative to the camera, projection adds perspective.)

2. The fly camera: yaw, pitch & a forward vector

How do we represent which way the camera looks? With two angles, the classic FPS approach:

  • Yaw — turning left/right, a rotation about the world's up axis.
  • Pitch — looking up/down, a rotation about the camera's right axis.

From those two angles we compute a unit forward vector (spherical to Cartesian):

forward = normalize( cos(pitch)·cos(yaw),
                     sin(pitch),
                     cos(pitch)·sin(yaw) )

Then viewMatrix() = lookAt(position, position + forward, worldUp). We clamp pitch to ±89°: at exactly ±90° the forward vector lines up with "up" and the camera gimbal-locks (the math for "right" collapses and the view flips).

3. Delta-time: motion that doesn't depend on frame rate

Naïvely, "move forward a bit each frame" makes a 600-FPS machine fly ten times faster than a 60-FPS one. The fix is delta-time (dt): the real elapsed seconds since the previous frame. Multiply speeds by it and everything is measured in units per second, identical on any machine:

distance_this_frame = speed_per_second · dt

We measure dt with SDL_GetTicksNS() (nanosecond precision) and clamp it — after a breakpoint or a long stall a giant dt would teleport the camera across the map.

4. Two kinds of input: events vs. polled state

SDL gives input two ways, and using the right one matters:

KindHowGood for
EventsSDL_PollEventDiscrete moments: window closed, Escape pressed, a mouse-motion delta.
Polled stateSDL_GetKeyboardStateContinuous questions: "is W held right now?"

If you moved the camera on key-down events, motion would stutter at your OS's key-repeat rate. So movement reads the live keyboard array every frame, while one-shot things (quit, look deltas) stay event-driven. A fly camera needs both.

5. Mouse look: relative mouse mode

For FPS-style looking, we don't want the cursor bumping into the screen edge. SDL's relative mouse mode (SDL_SetWindowRelativeMouseMode) hides the cursor, locks it to the window, and reports motion as deltas (xrel, yrel) no matter how far you move. We feed those deltas straight into yaw/pitch. (We enable it only for an interactive run, so the headless screenshot path doesn't grab your cursor.)

6. The code

The view matrix — src/math/Mat4.hpp

lookAt(eye, center, up) joins our math library, reusing the cross/normalize/dot from Vec.hpp. It's pure, so it's unit-tested in test_math.cpp (a camera at (0,0,5) looking at the origin maps the origin to (0,0,−5) in view space).

The camera — src/scene/Camera.{hpp,cpp}

Holds position, yaw, pitch; exposes viewMatrix(), processKeyboard(keys, dt) (WASD + E/Q), and addMouseLook(dx, dy). It lives in scene/, above the renderer: it knows nothing about the GPU, it just hands over a view matrix. Its logic is covered by test_camera.cpp (W moves along forward, pitch clamps, …).

Driving it — src/core/Engine.cpp

// once, for an interactive run:
SDL_SetWindowRelativeMouseMode(window, true);
Uint64 last = SDL_GetTicksNS();

while (running) {
    processEvents();                          // quit, Esc, mouse-motion → addMouseLook
    float dt = (SDL_GetTicksNS() - last) / 1e9f;   last = now;  (clamped)
    camera.processKeyboard(SDL_GetKeyboardState(nullptr), dt);  // continuous WASD
    renderer.renderFrame(clearColor, camera.viewMatrix());      // hand over the view
}

Drawing the cluster — GpuRenderer.cpp

The renderer now takes the view matrix. recordScene() binds the cube's geometry once, then for each position in a small hardcoded list it pushes that cube's Projection · View · Model and draws — the per-object uniform pattern (one set of buffers, many placements). Projection is rebuilt each frame from the swapchain's actual size, so resizing the window never distorts the image.

7. Controls

  • W / S — move forward / back
  • A / D — strafe left / right
  • E / Q — rise / descend
  • Mouse — look around
  • Esc — quit

8. Try it yourself

  1. Change the feel. Tweak moveSpeed_ and mouseSensitivity_ in Camera.hpp.
  2. Prove delta-time works. Temporarily multiply movement by a constant instead of dt, then toggle vsync / frame caps and feel the speed change with frame rate — the bug dt prevents.
  3. Break the pitch clamp. Remove the ±89° limit and look straight up — watch the view flip as it gimbal-locks.
  4. Add cubes. Extend kCubePositions in GpuRenderer.cpp and fly through your new scene.
